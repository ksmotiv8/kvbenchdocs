Setup: Bonsai

Bonsai 1-bit 8B on PrismML's llama.cpp

Bonsai uses a custom quantization format (Q1_0_g128) that is not in upstream llama.cpp. You need PrismML's fork. The model is public on HuggingFace, no access token required.

What you need

  • NVIDIA GPU with CUDA (any modern card works, 4+ GB VRAM)
  • cmake, gcc, git
  • huggingface-cli or hf for model download

Setup (one script)

bash scripts/setup-bonsai.sh

This script (source) clones the PrismML llama.cpp fork, builds with CUDA, and downloads the Bonsai-8B GGUF (1.15 GB).

Start the server

bash scripts/start-bonsai.sh

Starts on port 3001 by default. OpenAI-compatible API at http://localhost:3001/v1/chat/completions. On an L4, startup takes about 5 seconds and the model uses ~2 GB VRAM.

Verify it works

verify bonsai
$ curl -s http://localhost:3001/v1/chat/completions \ -H "Content-Type: application/json" \ -d '{"model":"default", "messages":[{"role":"user","content":"What is 2+2?"}], "max_tokens":16,"temperature":0}' | jq -r '.choices[0].message.content' 2 + 2 = 4.

Fork dependency. Bonsai only works on PrismML's llama.cpp fork. If upstream llama.cpp or SGLang add Q1_0_g128 support, this dependency goes away. Until then, you are running a vendor fork.

Setup: Gemma 4

Gemma 4 on vLLM

Gemma 4 launched April 1, 2026. At time of writing, SGLang does not support it yet (issue #21952 is open). vLLM v0.19.0 (released April 3) does support it, but requires a bleeding-edge transformers install because the gemma4 model type is too new for the latest PyPI release.

What you need

  • NVIDIA GPU with CUDA, 20+ GB VRAM for E2B, 21+ GB for E4B
  • Python 3.12, uv package manager

Setup

# E2B (2.3B effective params, 9.5 GB model, fits on L4)
GEMMA4_MODEL=E2B bash scripts/setup-gemma4.sh

# E4B (4.5B effective params, 14.9 GB model, tight on L4)
GEMMA4_MODEL=E4B bash scripts/setup-gemma4.sh

This script (source) creates a venv, installs vLLM 0.19.0, and installs transformers from source.

Start the server

GEMMA4_MODEL=E2B bash scripts/start-gemma4.sh

Starts on port 3000 by default. First run downloads the model from HuggingFace (~9.5 GB for E2B, ~14.9 GB for E4B). On an L4, the E2B uses ~19.9 GB VRAM. The E4B uses ~20.4 GB.

CUDA graphs not working yet. Neither SGLang nor vLLM supports CUDA graph capture for Gemma 4 at this time. All latency numbers include extra kernel launch overhead that will improve as runtime support matures. Expect 10-30% improvement once CUDA graphs work.

Setup: SGLang models

Setup commands for additional models

The following models all run under SGLang on the same L4. Each launch command below was taken from the actual experiment runs. The flags (--mem-fraction-static, --cuda-graph-max-bs, --max-running-requests) are tuned to fit VRAM on a single 24 GB L4 with a small batch size.

Qwen 2.5 7B AWQ

# Qwen 2.5 7B AWQ
python -m sglang.launch_server \
    --model-path /opt/models/qwen/Qwen2.5-7B-Instruct-AWQ \
    --host 0.0.0.0 --port 3000 \
    --mem-fraction-static 0.55 --context-length 4096 \
    --cuda-graph-max-bs 2 --max-running-requests 2

Non-reasoning baseline. CUDA graphs are enabled (--cuda-graph-max-bs 2) and give a measurable ms/tok improvement over eager mode. --mem-fraction-static 0.55 leaves headroom for a second model to run alongside on the same GPU.

Qwen 3.5 9B AWQ (reasoning model)

# Qwen 3.5 9B AWQ (reasoning model)
python -m sglang.launch_server \
    --model-path cyankiwi/Qwen3.5-9B-AWQ-4bit \
    --host 0.0.0.0 --port 3000 \
    --mem-fraction-static 0.55 --context-length 4096 \
    --cuda-graph-max-bs 2 --max-running-requests 2
# Note: enable_thinking=false in chat template via API to disable reasoning

Qwen 3.5 is a reasoning model by default. To compare it fairly against non-reasoning models, pass enable_thinking=false in the chat template arguments through the API. Otherwise it burns tokens on internal thinking before every answer.

Qwen 3.5 9B BF16

# Qwen 3.5 9B BF16 (full precision reference)
python -m sglang.launch_server \
    --model-path Qwen/Qwen3.5-9B-Instruct \
    --host 0.0.0.0 --port 3000 \
    --mem-fraction-static 0.85 --context-length 4096 \
    --cuda-graph-max-bs 2 --max-running-requests 2

Full-precision BF16 variant used to measure the quality cost of AWQ 4-bit. At 19 GB on disk and ~22 GB VRAM, this is a tight fit on an L4. Run it alone, not alongside other models.

GPT-OSS 20B

# GPT-OSS 20B (reasoning model, native MXFP4)
python -m sglang.launch_server \
    --model-path openai/gpt-oss-20b \
    --host 0.0.0.0 --port 3000 \
    --mem-fraction-static 0.85 --context-length 8192 \
    --cuda-graph-max-bs 4 --max-running-requests 4 \
    --reasoning-parser gpt-oss

Ships natively in MXFP4 so there is no extra quantization step. The --reasoning-parser gpt-oss flag splits the reasoning trace from the final answer in the API response. Without it, the reasoning block appears inline and breaks downstream parsing.

GLM 4.7 Flash AWQ

# GLM 4.7 Flash AWQ (reasoning model)
python -m sglang.launch_server \
    --model-path QuantTrio/GLM-4.7-Flash-AWQ \
    --host 0.0.0.0 --port 3000 \
    --mem-fraction-static 0.85 --context-length 8192 \
    --cuda-graph-max-bs 4 --max-running-requests 4 \
    --reasoning-parser glm45

Use --reasoning-parser glm45 (note: glm45, not glm47) to separate thinking from the final answer. Same gotcha as GPT-OSS: without the parser, the reasoning is returned as part of message.content and downstream tools see unparsed think blocks.

Reproduce

Run it yourself

Everything is in the inference-bench repo. The benchmark scripts work against any OpenAI-compatible API endpoint.

Full setup on a fresh GPU instance

# Clone
git clone https://github.com/ksmotiv8/kvbenchdocs.git
cd inference-bench

# Install Bonsai (builds llama.cpp, downloads 1.15 GB model)
bash scripts/setup-bonsai.sh

# Install Gemma 4 (installs vLLM, downloads model on first start)
GEMMA4_MODEL=E2B bash scripts/setup-gemma4.sh

# Start both servers
bash scripts/start-bonsai.sh          # port 3001
GEMMA4_MODEL=E2B bash scripts/start-gemma4.sh  # port 3000

# Run benchmarks
bash benchmarks/compare.sh 3000 google/gemma-4-E2B-it 3001 default

# Stop everything
bash scripts/stop-all.sh

Minimum hardware

  • Bonsai only: Any NVIDIA GPU with 4+ GB VRAM, 2 GB disk
  • Gemma 4 E2B only: 24 GB VRAM (L4, RTX 4090, A10G), 15 GB disk
  • Both side by side: 24 GB VRAM (tight, ~22 GB used), 20 GB disk
  • Gemma 4 E4B + Bonsai: 24 GB VRAM (very tight, ~22.3 GB used)

What the benchmark scripts measure

benchmarks/latency.sh (source) sends 8 prompts with a concise system prompt and measures total latency, token count, and ms/tok for each. It warms up the server before measuring.

benchmarks/quality.sh (source) sends prompts across factual knowledge, reasoning, and instruction following categories with greedy decoding. Each answer is auto-scored against known correct answers.

Both scripts take a port number and model name as arguments. They work with any server that exposes /v1/chat/completions.

Formal benchmarks

How to run IFEval and GSM8K yourself

# Install
pip install "lm-eval[api]"

# IFEval against any OpenAI-compatible server
lm_eval --model local-chat-completions \
  --model_args model=default,base_url=http://localhost:3001/v1,num_concurrent=1 \
  --tasks leaderboard_ifeval --batch_size 1

# GSM8K (chain-of-thought)
lm_eval --model local-chat-completions \
  --model_args model=default,base_url=http://localhost:3001/v1,num_concurrent=1 \
  --tasks gsm8k_cot --batch_size 1
Going deeper

Custom evals with Inspect AI

Standard benchmarks (IFEval, GSM8K) measure general capability. But the question driving this work is more specific: can these models function as components inside a larger system? That requires testing things standard benchmarks do not cover: multi-step pipelines, format compliance at scale, and tool calling.

Inspect AI is an eval framework built by the UK AI Safety Institute and used by Anthropic, DeepMind, and Grok. It supports composable solver pipelines, built-in tool use, sandboxed code execution, and publishable result bundles. It also natively supports all three of our runtimes (vLLM, SGLang, llama-cpp-python) through simple environment variables.

Three custom evals

Pipeline latency (source). Tests whether multi-step LLM pipelines produce correct output when chained. A 2-step pipeline classifies the input then generates with a targeted prompt. A 4-step pipeline classifies, plans, executes, and verifies. If the blog's thesis is right (low latency enables chained calls), these pipelines should work at interactive speed on the fast models and degrade on the slow ones. Inspect tracks token usage and timing per step, so we can see where the pipeline spends its time.

Format compliance (source). Our quick test found that Bonsai outputs raw JSON while Gemma 4 wraps it in markdown fences. Two prompts is not enough to make that claim. This eval tests 10 prompts across JSON output, executable Python, yes/no answers, and numeric-only responses. The scorer checks whether the output is machine-parseable without post-processing.

Tool calling (source). Tests whether models can use tools (weather lookup, math calculation, database search) when given tool definitions. Simpler than BFCLv3 but directly relevant: if low-latency inference enables agentic workflows, the model needs to call tools correctly. Inspect's built-in tool support makes this a straightforward eval.

Setup

# Install
pip install inspect-ai

# Configure for your runtime
export LLAMA_CPP_PYTHON_BASE_URL=http://localhost:3001/v1    # Bonsai
export VLLM_BASE_URL=http://localhost:3000/v1                 # Gemma 4
export SGLANG_BASE_URL=http://localhost:3000/v1               # Qwen

# Run
inspect eval evals/pipeline_latency.py --model llama-cpp-python/bonsai
inspect eval evals/format_compliance.py --model vllm/google/gemma-4-E2B-it
inspect eval evals/tool_calling.py --model sglang/default

# View results (generates static HTML)
inspect view bundle --output results/

Results queued. These evals will run after the IFEval/GSM8K benchmarks complete. The eval files are in the repo and can be run independently against any OpenAI-compatible server.
