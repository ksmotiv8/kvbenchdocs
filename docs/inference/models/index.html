What Low-Latency Inference Unlocks

Eight models tested on a single L4. The fastest one is fast enough to put an LLM somewhere you would not have considered before.

Thesis

Latency is what determines where an LLM can sit

Most LLM benchmarks tell you which model is smarter. That is the wrong question if you are putting one inside a system. The right question is where in the system can it sit, and the answer is set by latency.

At 2 seconds per response, an LLM is a destination. The user clicks something and waits. At 200 milliseconds, it can sit in a request handler that returns a rendered page. At 50 milliseconds, it can sit inside a streaming pipeline that processes events without anyone noticing. Each threshold opens a different set of architectures. Most published benchmarks ignore the threshold entirely and report ms/tok averaged over compute-rich datacenter hardware.

This post tests how low latency can go on a single NVIDIA L4 (24 GB, ~$0.80/hr) and what changes at each step. Eight models, three runtimes, real wall-clock numbers. Setup commands and detailed validation are on companion pages.

Why Bonsai is interesting

One model crosses a threshold

Of the eight models tested, Bonsai 1-bit 8B is the only one under 10 ms/tok. At 8.4 ms/tok, a 50-token response completes in around 420 ms end-to-end including network. Everything else in the set is between 19 and 70 ms/tok, which puts a 50-token response in the 1-3.5 second range. The gap is not subtle.

The mechanical reason is straightforward. Bonsai weights are 1.15 GB on disk, against 5.2 GB for Qwen 2.5 7B at INT4 and 9.5 GB for Gemma 4 E2B at BF16. Token generation on a memory-bandwidth-bound GPU like the L4 is gated by how much data you have to read per token. Less weight data, faster tokens.

The cost is real. Bonsai scored 68.9% on GSM8K against 93.0% for its unquantized base model (Qwen3-8B per the Bonsai whitepaper). That is a 24-point regression on multi-step math. It scored 82.8% on IFEval, second only to Gemma 4 E4B at 86.9% among non-reasoning models, and it produced raw JSON and executable code without markdown wrapping in every test prompt I ran. Aggressive quantization preserves instruction following and structured output much better than it preserves arithmetic.

That tradeoff matters because of what 8 ms/tok lets you do.

What it unlocks

Sub-500ms inference is a different building block

The interesting use cases are not "fast chatbot." They are the cases where an LLM call has to fit inside something else's budget.

  • Inline content moderation. Every message in a chat or comment thread runs through a classifier before it is delivered. At 2 seconds per call this is an offline batch job. At 400 ms it is part of the message delivery path and the user does not notice.
  • Structured extraction in a logging pipeline. Log lines hit an LLM that emits a JSON event with severity, component, and root-cause hypothesis. The Bonsai-style raw JSON output matters here because the next stage is a database write, not a human reader.
  • Streaming function calls. Voice assistants and conversational agents that need to invoke a tool before the user finishes their sentence. At 8 ms/tok the model can produce a tool call before a human-paced speaker has finished a clause.
  • Search query rewriting. Every query gets rewritten by an LLM (clarify intent, expand abbreviations, add filters) before hitting the search backend. Adding 50 ms to a search request is acceptable. Adding 1.5 s is not.
  • Inline code review on edit. Lint-style feedback from an LLM that runs on every save in an IDE without noticeable delay. This only works at the lower end of the ms/tok range.

None of these use cases need a frontier model. They need a model that gives the right answer reliably within a tight latency budget. The decision is bounded by what the system can absorb, not what produces the highest leaderboard score. Once you are below the threshold, model selection becomes a quality and format-compliance question instead of a latency question.

The middle

Everything else lives in a 19-21 ms/tok cluster

Four of the eight models (Qwen 2.5 7B AWQ, Gemma 4 E2B, GLM 4.7 Flash AWQ, GPT-OSS 20B) sit between 19 and 21 ms/tok. At this band, model selection is about quality, format behavior, and runtime stability rather than raw speed.

  • Qwen 2.5 7B AWQ is the boring stable choice. SGLang has supported it cleanly for months, CUDA graphs work, behavior is predictable. Lower benchmark scores than the newer models but no surprises in deployment.
  • Gemma 4 E2B hits 81% IFEval at 2.3B effective parameters, which is impressive on paper. It still uses around 20 GB of VRAM on the L4 because of its embedding layout, so the "edge" framing in the marketing is generous. It wraps output in markdown fences, which limits its use as a pipeline component.
  • GLM 4.7 Flash AWQ and GPT-OSS 20B are reasoning models. They show 20-21 ms/tok in the chart but that number measures decode during their internal thinking phase. In our 48-token latency test, every reasoning-model response had an empty visible content field. Real time-to-first-visible-token is several seconds. They post the best benchmark scores in the set (GLM 91.3% on GSM8K, the highest) but they do not belong in the same latency conversation as the rest.

Gemma 4 E4B at 40.5 ms/tok and Qwen 3.5 9B BF16 at 68.5 ms/tok are slower options. E4B buys ~4 IFEval points over E2B for roughly double the latency. Qwen 3.5 BF16 is impractical on a single L4 without running into memory and time limits.

Caveats

What this comparison is and is not

  • Each model runs on a different runtime (llama.cpp for Bonsai, SGLang for Qwen and reasoning models, vLLM for Gemma 4). Cross-model latency comparisons are directional, not absolute. Same-model-different-runtime tests would isolate this confound; no single runtime supports every format in this set.
  • All numbers are single-request latency. Multi-user serving has different bottlenecks.
  • Bonsai's 24-point GSM8K regression vs its base model is a real cost, not an aside. The 1-bit format is excellent for short structured output and noticeably weaker on multi-step arithmetic.
  • Reasoning model ms/tok measures decode during the thinking phase, not wall-clock time to a visible answer. In our 48-token test the visible content field was empty for every reasoning-model response.
  • Qwen 3.5 9B IFEval and GSM8K results are harness artifacts, not real capability. lm-eval cannot pass enable_thinking=false through the chat-completions API, so the model burns its budget inside <think> blocks and never reaches the answer. The BF16 IFEval was killed after 12 hours of projected runtime.
  • CUDA graphs are not yet supported for Gemma 4 on vLLM, so the Gemma numbers include extra kernel launch overhead the others avoid.
  • Reasoning model IFEval and GSM8K were limited to 150 prompts each (vs 541 / 1319 for non-reasoning models) due to time constraints.
