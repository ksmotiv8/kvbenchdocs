How low can LLM inference latency go on a single L4?

Eight models, three runtimes, one GPU. What I learned about quantization, speculative decoding, and reasoning models when latency is the binding constraint.

Motivation

Latency determines where an LLM can sit

At Momento, I spend my days obsessing over microsecond-level latency in caches and message buses. That work has given me an intuition that I keep bringing to LLM inference: latency is not just a number you optimize. It determines what systems you can build. Cross a threshold and entire architectures that were previously off the table become viable.

At 2 seconds per response, an LLM is a destination. The user clicks something and waits. At 500 milliseconds, it can sit inside a request handler that returns a rendered page. At 50 milliseconds, it can sit inside a streaming pipeline that processes events without anyone noticing. These thresholds are not incremental improvements. They are different classes of system.

Most published benchmarks report ms/tok averaged over datacenter hardware. That is the wrong answer to the wrong question if what you care about is where in a system an LLM can live. The right question is: given the hardware I can actually afford and the latency budget my system can actually absorb, which model fits?

This post tests how low latency can go on a single NVIDIA L4 (24 GB VRAM, roughly 300 GB/s memory bandwidth, about $0.80/hr). I started with a narrow question (does speculative decoding help as much as the papers claim?) and the scope expanded as the weeks went on. It now covers eight models across three runtimes.

The short version: one model crosses the threshold that matters. Everything else lives in a 19-21 ms/tok cluster where the interesting choices are about quality, output format, and whether you are willing to accept a reasoning model's hidden latency cost.

The hardware

Why the L4

I chose the L4 because the constraints are honest. An H100 or H200 will run anything you throw at it, which makes bottleneck analysis harder, not easier. The L4 has modest FP16 throughput and modest memory bandwidth, which means every optimization decision shows up in the numbers immediately.

The key technical point: single-request token generation is memory-bandwidth-bound. Each output token requires reading essentially the full weight matrix from VRAM. On the L4, that means smaller weights translate directly to faster tokens. This mechanism is responsible for most of the results that follow.

The L4 is also a starting point, not a destination. Everything I measure here gets better on an H200 with HBM3e. But I wanted to understand the fundamentals on hardware where every millisecond of overhead is visible.

Experiment 1

SGLang configurations: does speculative decoding help?

I started with SGLang, which consistently posts the lowest single-request latencies I have measured across open-source engines. My first goal was to figure out whether speculative decoding actually helps in the latency regime I care about.

Speculative decoding tries to solve a real problem: token generation is sequential, each token requires reading the full weight matrix, and latency is the cost. The idea is to use a small draft model to guess several tokens ahead, then verify them in a single forward pass of the main model. If the guesses are right, you get multiple tokens for the cost of one big pass. The catch is that the draft model adds its own overhead, and wrong guesses waste work.

I tested six configurations with the same setup every time: concise system prompt, short outputs (10-18 tokens), single-request.

The smallest model won. Qwen 1.5B at FP16, no speculation, 125ms total. That was not the result I expected. I had assumed that a 7B model with a well-tuned draft model would come close on latency and win on quality. It did not. The draft model overhead on short outputs was never recovered.

Speculation has a fixed cost per call. On short outputs (10-13 tokens), that fixed cost dominates. STANDALONE with a 0.5B draft added about 40ms per request. On the Llama AWQ + EAGLE3 rows, you can see what speculation actually looks like when it works: the ms/tok drops from 13.5 to 12.9 when the output grows from 18 to 64 tokens, because the per-token savings compound. For genuinely short answers, speculation is an anti-optimization.

Quantization is the effect that matters. The best ms/tok (12.9) and best VRAM (6.2 GB) both come from the AWQ INT4 Llama row, not from speculation. AWQ cuts weight size by roughly 4x from FP16, which is where the real latency gain comes from. Speculation was a small adjustment on top of that.

Experiment 2

Then I tried the 1-bit model

I pulled in 1-bit Bonsai 8B out of curiosity. It is an 8B model (built from Qwen3-8B) where every weight is stored as a single sign bit with a shared FP16 scale per 128-weight group. That is 1.125 effective bits per weight, down from 16 in FP16, so the model fits in 1.15 GB on disk. It runs on PrismML's llama.cpp fork. I cloned it, built with CUDA, downloaded the GGUF, and started a server on port 3001 alongside SGLang on 3000.

I ran the same set of prompts against both and measured wall-clock latency. The 1-bit 8B against the FP16 1.5B felt like an unfair comparison to set up, but the results were clarifying.

Bonsai generates tokens at 8.4 ms/tok. That is 40% faster per token than the 1.5B FP16 model on the same GPU, and the model has 5.5x as many parameters. The mechanical reason is simple: the L4 is memory-bandwidth-bound, Bonsai's weights are roughly a quarter the size of Qwen 1.5B's even before considering the parameter count, and smaller weights mean less data to move per token.

On 48-token latency tests, the per-token speed compounds: a typical Bonsai response came in at 100-150ms total against 230ms for the 1.5B baseline. Over 8 test prompts, Bonsai won 7. The one loss happened because Bonsai chose to write a longer answer (27 tokens vs 11), not because it generated more slowly.

There is a real cost. Bonsai scored 68.9% on GSM8K against 93.0% for its unquantized base model (Qwen3-8B per the Bonsai whitepaper). That is a 24-point regression on multi-step math. It scored 82.8% on IFEval, which is higher than any other non-reasoning model I tested except Gemma 4 E4B. Aggressive quantization preserves instruction following and structured output much better than it preserves arithmetic.

Bonsai also produced raw JSON and executable Python in every format-sensitive prompt I tested, without wrapping the output in markdown fences. I did not expect that to matter as much as it does. It turns out to matter a lot, which I will come back to after the rest of the model results.

Caveat on the runtime: Bonsai runs on PrismML's vendor fork of llama.cpp, not upstream. That affects the latency comparison in two ways. First, llama.cpp has lower per-request overhead than SGLang for single-stream workloads, so some of the gap may be runtime rather than model. Second, relying on a single-vendor fork in production carries its own risks. The 8.4 ms/tok number is real, but it is not a same-runtime comparison.

Experiment 3

Opening the aperture: Gemma 4, Qwen 3.5, GPT-OSS, GLM 4.7

Once Bonsai was set up, I added Gemma 4 (E2B and E4B, Google's per-layer-embedding dense models), Qwen 2.5 7B AWQ (as a stable SGLang baseline), Qwen 3.5 9B in AWQ and BF16, GPT-OSS 20B (OpenAI's MoE with native MXFP4), and GLM 4.7 Flash AWQ (Zhipu's lightweight MoE). I ran the same 8-prompt latency test against each, plus IFEval (541 prompts), GSM8K (1319 problems for non-reasoning models, 150 for reasoning models due to time constraints), and a 12-prompt manual quality suite.

The cluster in the middle of the chart (Qwen 2.5, GLM, Gemma E2B, GPT-OSS, all between 19 and 21 ms/tok) is not a coincidence. Observations

What the numbers are telling us

One model is actually fast. Bonsai at 8.4 ms/tok is the only option in the set under 10. Everything else is at least 2.3x slower. The gap is not a few percent at the edges, it is the difference between "this can sit in a request path" and "this is a destination the user waits for."

Quantization beats speculation for short outputs. On the L4, INT4 AWQ saves more latency than speculative decoding does. The AWQ row beat the AWQ + EAGLE3 row on total time for short outputs, because speculation has a fixed cost per call that short answers do not amortize. Speculation is a long-output optimization, not a general latency tool.

Reasoning models hide their latency cost behind a misleading number. GPT-OSS 20B, GLM 4.7 Flash AWQ, and Qwen 3.5 all show a decode rate in the 20-70 ms/tok band, but the number measures decode during the model's internal thinking phase. In our 48-token latency test, the visible content field was empty in every response for every reasoning model. The model spends the entire budget generating <think> tokens that the user never sees. Real time-to-first-visible-token runs several seconds. GLM 4.7 Flash AWQ has the best math score in the set at 91.3% on GSM8K, and it pays a latency cost users will notice.

Format behavior splits the set cleanly. Asked for raw JSON, Bonsai outputs {"name":"Qwen","language":"Chinese"}. Every other model wraps it in a markdown code fence. Asked for an executable Python function, Bonsai outputs the function. The others wrap it. This is suggestive rather than definitive (a small number of prompts), but the pattern was consistent. For use cases where the output feeds into another system without a human in the loop, format literalism is more valuable than a few percentage points of benchmark score.

Factual recall does not correlate with size or generation. I asked every model "what does AWQ stand for in the context of LLM quantization?" The correct answer is Activation-aware Weight Quantization. Three models got it right (Gemma 4 E2B, Gemma 4 E4B, GLM 4.7 Flash AWQ). Three got it wrong with three different wrong answers (Bonsai: "Approximate," Qwen 2.5: "Adaptive," GPT-OSS 20B: "Adaptive"). The split does not correlate with parameter count, quantization, or model generation. Knowing the right facts is not a property that scales cleanly.

What it unlocks

Sub-500ms inference is a different building block

The interesting use cases for 8 ms/tok are not "snappier chatbot." They are the cases where an LLM has to fit inside something else's latency budget without adding a noticeable delay.

  • Inline content moderation. Every message in a chat or comment thread runs through a classifier before delivery. At 2 seconds per call this is an offline batch job. At 400ms it is part of the message delivery path and the user does not notice.
  • Structured extraction on log streams. Log lines hit an LLM that emits a JSON event with severity, component, and root-cause hypothesis. The next stage is a database write, not a human reader, so Bonsai's markdown-free JSON output matters directly.
  • Streaming function calls in voice agents. The model needs to produce a tool call before the human speaker has finished the clause. This only works at the low end of the ms/tok range.
  • Search query rewriting in the request path. Every query gets expanded and clarified by an LLM before hitting the search backend. Adding 50ms is acceptable. Adding 1.5 seconds is not.
  • Inline code review on save. Lint-style feedback from an LLM that runs on every editor save without perceptible delay.

There is a corollary worth calling out. If inference drops to 100ms, every other system in the pipeline has to match. A 100ms LLM call followed by a 500ms vector search is still a 600ms round trip. Low-latency inference inevitably demands low-latency RAG, context retrieval, and tool execution. The model stops being the bottleneck and the infrastructure around it becomes the bottleneck. That is the problem space I work in at Momento, and it is where a lot of the interesting systems work is going to happen in the next year.

Decision guide

When to use which model

The decision is bounded by what your system can absorb, not what produces the highest leaderboard score. Here is how I think about the choice after running these tests.

Bonsai 1-bit 8B. Pick this when latency is the binding constraint and the work is short-form. Good fit: classification, prompt rewriting, structured extraction, format normalization, tool call generation. Do not rely on it for multi-step arithmetic.

Qwen 2.5 7B AWQ. The boring stable default. SGLang has supported it cleanly for months, CUDA graphs work, deployment behavior is predictable. Lower benchmark scores than newer models but no surprises in production.

Gemma 4 E2B or E4B. Pick when knowledge density matters more than raw speed. E2B at 81% IFEval on 2.3B effective parameters is impressive on paper, though it still uses around 20 GB of VRAM on the L4 because of its embedding layout. E4B buys roughly 4 IFEval points over E2B for double the decode cost. Both wrap output in markdown, which limits their use as pipeline components.

GLM 4.7 Flash AWQ or GPT-OSS 20B. Reasoning models. Pick when answer quality matters more than time-to-first-token and your system can stream the visible answer when it eventually arrives. GLM wins the formal benchmarks (91.3% GSM8K, the best in this comparison). GPT-OSS has shorter reasoning traces and therefore better end-to-end wall-clock time for equivalent work. Do not use either as a pipeline subroutine.

Summary. Bonsai for short-form structured output. Gemma 4 for instruction following and knowledge density. Qwen 2.5 as a stable production default. GLM 4.7 Flash AWQ when math quality matters. GPT-OSS 20B as a faster-responding reasoning alternative. None of them is universally best.

Caveats

What this comparison is and is not

  • Each model runs on a different runtime (llama.cpp for Bonsai on PrismML's vendor fork, SGLang for Qwen and the reasoning models, vLLM for Gemma 4) with different optimization levels. Take the cross-model latency comparisons as directional, not absolute. Same-model-different-runtime tests would isolate this confound; no single runtime supports every format in the set.
  • All numbers are single-request latency. Multi-user serving has different bottlenecks.
  • Bonsai's 24-point GSM8K regression vs its base model (68.9% vs Qwen3-8B's reported 93.0%) is a real cost, not an aside. The 1-bit format is excellent for short structured output and noticeably weaker on multi-step arithmetic.
  • Reasoning model ms/tok numbers (GPT-OSS, GLM 4.7, Qwen 3.5) measure decode during the thinking phase, not wall-clock time to a visible answer. In our 48-token test the visible content field was empty for every reasoning-model response.
  • Qwen 3.5 9B IFEval and GSM8K results are harness artifacts, not real capability. lm-eval cannot pass enable_thinking=false through the chat-completions API, so the model burns its budget inside <think> blocks. The BF16 IFEval was killed after 12 hours of projected runtime.
  • CUDA graphs are not yet supported for Gemma 4 on vLLM, so the Gemma numbers include kernel launch overhead the others avoid.
  • Reasoning model IFEval and GSM8K were limited to 150 prompts each (vs 541 and 1319 for the non-reasoning models) due to time constraints.
  • The 12-prompt manual quality test is small. The format compliance observation in particular is based on two prompts and is suggestive rather than definitive.
What's next

Open questions

  • Gemma 4 26B-A4B MoE on H200. The MoE variant with 3.8B active params per token is the most architecturally interesting model I did not get to test. It needs 48 GB VRAM at BF16.
  • Quantized Gemma 4. Running E4B at INT4 or FP8 would cut the model size and should meaningfully improve ms/tok. AWQ and GPTQ variants are starting to appear but SGLang/vLLM support is still stabilizing.
  • Same-runtime comparison. Running Bonsai and Gemma 4 on the same engine would isolate the model-level latency difference from the runtime overhead.
  • Longer outputs. All my tests used short outputs (10-48 tokens). Speculative decoding should look better at longer output lengths, and the reasoning model latency story would look very different on longer answers where the thinking overhead is amortized.
  • Low-latency RAG. The corollary I mentioned earlier: once inference is fast, the retrieval layer needs to match.
