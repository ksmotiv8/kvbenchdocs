Bonsai 1-bit and Gemma 4

Two new model architectures for low-latency inference, tested on the same GPU. Different runtimes, different quantization, same prompts. How to set them up, how I validated quality, what the results mean, and how to reproduce everything yourself.

Context

Why these two models

I have been benchmarking low-latency inference on a single NVIDIA L4 (full writeup here). After testing six SGLang configurations, I wanted to try two models that take fundamentally different approaches to the latency problem.

Bonsai 1-bit 8B by PrismML attacks latency through extreme weight compression. Every weight is stored as a single sign bit with one FP16 scale per 128 weights. An 8B model fits in 1.15 GB. The thesis: if token generation is memory-bandwidth-bound, make the weights as small as physically possible.

Gemma 4 by Google DeepMind attacks latency through architectural efficiency. The E2B and E4B variants use Per-Layer Embeddings to maximize capability per parameter. The MoE variant activates only 3.8B of 26B parameters per token. The thesis: make the model smarter per unit of compute, not just smaller.

Both are Apache 2.0 licensed. Both claim to be deployable on edge hardware. I wanted to see how they compare on real latency, real quality, and real setup complexity.

Setup: Bonsai

Bonsai 1-bit 8B on PrismML's llama.cpp

Bonsai uses a custom quantization format (Q1_0_g128) that is not in upstream llama.cpp. You need PrismML's fork. The model is public on HuggingFace, no access token required.

What you need

  • NVIDIA GPU with CUDA (any modern card works, 4+ GB VRAM)
  • cmake, gcc, git
  • huggingface-cli or hf for model download

Setup (one script)

bash scripts/setup-bonsai.sh

This script (source) clones the PrismML llama.cpp fork, builds with CUDA, and downloads the Bonsai-8B GGUF (1.15 GB).

Start the server

bash scripts/start-bonsai.sh

Starts on port 3001 by default. OpenAI-compatible API at http://localhost:3001/v1/chat/completions. On an L4, startup takes about 5 seconds and the model uses ~2 GB VRAM.

Verify it works

verify bonsai
$ curl -s http://localhost:3001/v1/chat/completions \ -H "Content-Type: application/json" \ -d '{"model":"default", "messages":[{"role":"user","content":"What is 2+2?"}], "max_tokens":16,"temperature":0}' | jq -r '.choices[0].message.content' 2 + 2 = 4.

Fork dependency. Bonsai only works on PrismML's llama.cpp fork. If upstream llama.cpp or SGLang add Q1_0_g128 support, this dependency goes away. Until then, you are running a vendor fork.

Setup: Gemma 4

Gemma 4 on vLLM

Gemma 4 launched April 1, 2026. At time of writing, SGLang does not support it yet (issue #21952 is open). vLLM v0.19.0 (released April 3) does support it, but requires a bleeding-edge transformers install because the gemma4 model type is too new for the latest PyPI release.

What you need

  • NVIDIA GPU with CUDA, 20+ GB VRAM for E2B, 21+ GB for E4B
  • Python 3.12, uv package manager

Setup

# E2B (2.3B effective params, 9.5 GB model, fits on L4)
GEMMA4_MODEL=E2B bash scripts/setup-gemma4.sh

# E4B (4.5B effective params, 14.9 GB model, tight on L4)
GEMMA4_MODEL=E4B bash scripts/setup-gemma4.sh

This script (source) creates a venv, installs vLLM 0.19.0, and installs transformers from source.

Start the server

GEMMA4_MODEL=E2B bash scripts/start-gemma4.sh

Starts on port 3000 by default. First run downloads the model from HuggingFace (~9.5 GB for E2B, ~14.9 GB for E4B). On an L4, the E2B uses ~19.9 GB VRAM. The E4B uses ~20.4 GB.

CUDA graphs not working yet. Neither SGLang nor vLLM supports CUDA graph capture for Gemma 4 at this time. All latency numbers include extra kernel launch overhead that will improve as runtime support matures. Expect 10-30% improvement once CUDA graphs work.

Validation

How I tested quality

I ran 12 prompts across four categories: factual knowledge, mathematical reasoning, code correctness, and instruction following. All prompts have objectively verifiable answers. This is a small test suite. It is useful for spotting obvious quality differences, but 12 questions cannot substitute for a proper benchmark. The IFEval and GSM8K results below provide more statistical weight.

The test suite

bash benchmarks/quality.sh 3001 default    # Bonsai
bash benchmarks/quality.sh 3000 google/gemma-4-E2B-it  # Gemma 4

The quality script (source) runs greedy decoding (temp=0) and auto-scores each answer.

Factual knowledge: what does AWQ stand for?

This question turned out to be the sharpest discriminator. The correct answer is "Activation-aware Weight Quantization." Gemma 4 (both E2B and E4B) got it right. Bonsai said "Approximate Weight Quantization," which is wrong. Qwen 2.5 7B AWQ said "Adaptive Weight Quantization," also wrong. Three different wrong answers from three different model families. Only Gemma 4 got it right at every size tested.

Code correctness: executable output matters

I asked both models to write a Fibonacci function and a string reversal function, then executed the output. Bonsai produced clean, executable Python on both code prompts I tested. Gemma 4 and Qwen wrapped their code in markdown triple-backtick fences (```python ... ```), which is correct for documentation but fails python3 execution. Two prompts is a small sample, but if your pipeline feeds LLM output directly into an interpreter, it is worth testing this behavior with your own prompts.

Instruction following: JSON output

I asked for raw JSON with specific keys. Bonsai output {"name":"Qwen","language":"Chinese"}, which passes json.loads(). Gemma 4 wrapped it in a markdown code fence, which fails parsing. Same pattern in my tests: Gemma 4 and Qwen produced human-readable formatting. Bonsai produced machine-parseable output. Two format-sensitive prompts is a small sample. Test this with your own prompts before relying on it.

Qwen 2.5 7B AWQ on SGLang is the interesting middle ground. At 19.1 ms/tok with CUDA graphs enabled, it is the fastest conventional (non-1-bit) model. It uses 13.2 GB VRAM, leaving room for a second model alongside it. SGLang is the most mature runtime of the three, with full CUDA graph support, RadixAttention, and established tooling. If you need a production-ready setup today without custom forks or bleeding-edge runtimes, Qwen on SGLang is the practical choice.

Why Bonsai is faster despite having more parameters

Token generation on the L4 is memory-bandwidth-bound. Each token requires reading the full weight matrix from VRAM. The L4 has ~300 GB/s of memory bandwidth. Bonsai's weights are 1.15 GB (1-bit with FP16 group scales). Gemma 4 E2B's weights are 9.5 GB at BF16. That is an 8x difference in data per token. The bandwidth wall hits Gemma 4 much harder.

There is an important caveat: these models run on different runtimes (llama.cpp vs vLLM). The runtimes have different overhead profiles. I did not isolate how much of the speed difference comes from the runtime vs the model. A proper controlled comparison would run both on the same engine, which is not possible today because neither runtime supports both model formats.

When to use which model

The right model depends on where it sits in your system and what it needs to do. Here is how I think about the choice after running these tests. One important caveat: each model runs on a different runtime (llama.cpp, SGLang, vLLM) with different optimization levels. Some of the latency differences come from the runtime, not the model. Take the ms/tok comparisons as directional, not absolute.

Bonsai 1-bit 8B: pipeline infrastructure. Use it when inference is a subroutine inside a larger system. Prompt classification, rewriting, routing, tool call generation, structured output. It is the fastest model I tested (8.4 ms/tok), uses the least VRAM (~2 GB), and its outputs are machine-parseable without post-processing. It does not wrap code in markdown fences. It does not add explanatory text when you ask for JSON. If your system chains multiple LLM calls and needs sub-200ms round trips, Bonsai is the only model I tested that gets close on an L4 (170ms average). The tradeoff is factual accuracy: it gets some knowledge questions wrong that both Gemma variants get right.

Gemma 4 E2B: knowledge-dense tasks on constrained hardware. Use it when you need factual accuracy from a small model that runs on edge hardware. E2B scored 5/5 on factual knowledge in my tests despite having only 2.3B effective parameters. Google's benchmarks show 60% on MMLU Pro, which is strong for a model this size. It runs on a Raspberry Pi 5 at 7.6 tok/s. On an L4, it runs at 20.8 ms/tok. The tradeoff is VRAM: the Per-Layer Embeddings inflate total size to 9.5 GB BF16, which means it uses 19.9 GB on vLLM. That is a lot of VRAM for a 2.3B-effective model. It also wraps outputs in markdown formatting, which is a problem for pipeline use cases.

Gemma 4 E4B: complex reasoning where quality matters more than speed. Google's benchmarks show E4B at 42.5% on AIME 2026 math (vs 37.5% for E2B), 58.6% on GPQA Diamond (vs 43.4%), and 52% on LiveCodeBench (vs 44%). The 10-15 point gap on hard reasoning tasks is real. In my simpler test suite, that gap did not surface (both scored 10/12), but my prompts were not hard enough to discriminate. If your use case involves multi-step reasoning, graduate-level questions, or competitive coding, E4B is worth the latency cost (40.5 ms/tok). If your prompts are simple classification or short-form QA, E2B gives the same answers faster.

Qwen 2.5 7B AWQ on SGLang: production-ready middle ground. Use it when you need a proven model on a mature runtime. SGLang has full CUDA graph support, RadixAttention, and continuous batching. Qwen 2.5 7B AWQ at INT4 runs at 19.1 ms/tok with 13.2 GB VRAM, leaving room for a second model alongside it. The tooling is stable: no custom forks, no bleeding-edge transformers installs. The tradeoff is that it scored lowest on quality (9/12) because it wraps code and JSON in markdown fences (like Gemma 4) and got AWQ wrong (like Bonsai). But 19 ms/tok with CUDA graphs is a strong baseline for latency-sensitive serving.

Summary. Bonsai for speed and format compliance. E2B for factual accuracy on edge. Qwen 2.5 7B AWQ for production readiness on SGLang. E4B for hard reasoning where you can afford 40+ ms/tok. None of them is universally best.

Reproduce

Run it yourself

Everything is in the inference-bench repo. The benchmark scripts work against any OpenAI-compatible API endpoint.

Full setup on a fresh GPU instance

# Clone
git clone https://github.com/ksmotiv8/kvbenchdocs.git
cd inference-bench

# Install Bonsai (builds llama.cpp, downloads 1.15 GB model)
bash scripts/setup-bonsai.sh

# Install Gemma 4 (installs vLLM, downloads model on first start)
GEMMA4_MODEL=E2B bash scripts/setup-gemma4.sh

# Start both servers
bash scripts/start-bonsai.sh          # port 3001
GEMMA4_MODEL=E2B bash scripts/start-gemma4.sh  # port 3000

# Run benchmarks
bash benchmarks/compare.sh 3000 google/gemma-4-E2B-it 3001 default

# Stop everything
bash scripts/stop-all.sh

Minimum hardware

  • Bonsai only: Any NVIDIA GPU with 4+ GB VRAM, 2 GB disk
  • Gemma 4 E2B only: 24 GB VRAM (L4, RTX 4090, A10G), 15 GB disk
  • Both side by side: 24 GB VRAM (tight, ~22 GB used), 20 GB disk
  • Gemma 4 E4B + Bonsai: 24 GB VRAM (very tight, ~22.3 GB used)

What the benchmark scripts measure

benchmarks/latency.sh (source) sends 8 prompts with a concise system prompt and measures total latency, token count, and ms/tok for each. It warms up the server before measuring.

benchmarks/quality.sh (source) sends prompts across factual knowledge, reasoning, and instruction following categories with greedy decoding. Each answer is auto-scored against known correct answers.

Both scripts take a port number and model name as arguments. They work with any server that exposes /v1/chat/completions.

Formal benchmarks

IFEval and GSM8K

The quality tests above are useful for quick comparisons but they are not standard benchmarks. To give these results more weight, I ran two widely-used evaluation suites using lm-evaluation-harness against the models served via their respective APIs.

About the benchmarks

IFEval (Instruction Following Evaluation) tests whether a model can follow precise formatting constraints: "write exactly 3 paragraphs," "include the word 'elephant' at least twice," "respond in all caps." It has 541 prompts, each with one or more verifiable constraints. Scoring is purely rule-based with no LLM judge. The metric is the average of prompt-level strict accuracy and instruction-level strict accuracy. This benchmark matters for pipeline use cases where the model must follow format instructions exactly.

GSM8K (Grade School Math 8K) tests mathematical reasoning with 1,319 word problems at roughly middle-school difficulty. The model generates a chain-of-thought solution and a final numeric answer. Scoring extracts the last number in the response and compares it to the ground truth. No LLM judge needed. This benchmark matters because math reasoning is one of the first capabilities to degrade under aggressive quantization.

Both benchmarks hit the model via the OpenAI-compatible chat completions API. The evaluation framework runs on CPU with no GPU contention. Each model was tested one at a time to ensure clean measurements.

Bonsai 1-bit 8B leads on instruction following. IFEval 82.8% (prompt strict 79.9%, instruction strict 85.7%). This is a notable result: the 1-bit model scores higher than both the BF16 Gemma 4 and the INT4 Qwen on a 541-prompt rule-based benchmark. It aligns with what we saw in the quick tests: Bonsai follows format instructions literally, which is exactly what IFEval measures.

GSM8K tells a different story. Bonsai scored 68.9% on math reasoning. Qwen 2.5 7B AWQ scored 70.6%. The gap is small but Bonsai does not dominate here the way it does on instruction following. Math reasoning is one of the capabilities most sensitive to quantization. A 1-bit model holding within 2 points of an INT4 7B model on 1,319 math problems is respectable, but it is not a clear win.

Gemma 4 E2B: IFEval 81.0%, close to Bonsai despite having far fewer parameters (2.3B effective vs 8.19B). GSM8K 63.7%, the lowest of the three. Math reasoning at 2.3B effective parameters is limited. For reference, Google's own benchmarks show E4B at 42.5% on AIME 2026, so math is not E2B's strength.

Gemma 4 E4B: IFEval 86.9%, the highest in this comparison and a meaningful jump over E2B's 81.0%. GSM8K 70.2%, matching Qwen 2.5 7B AWQ. The extra parameters (4.5B effective vs 2.3B) clearly help on both instruction following and math. The cost is 40.5 ms/tok, roughly 5x slower than Bonsai.

For context, Qwen3-8B (the base model Bonsai was built from) scores 81.5% on IFEval and 93.0% on GSM8K per the Bonsai whitepaper. Those scores are from an unquantized model on H100. The Qwen model tested above is Qwen 2.5 (an older generation, late 2024). Qwen 3.5 9B benchmarks are running next.

Going deeper

Custom evals with Inspect AI

Standard benchmarks (IFEval, GSM8K) measure general capability. But the question driving this work is more specific: can these models function as components inside a larger system? That requires testing things standard benchmarks do not cover: multi-step pipelines, format compliance at scale, and tool calling.

Inspect AI is an eval framework built by the UK AI Safety Institute and used by Anthropic, DeepMind, and Grok. It supports composable solver pipelines, built-in tool use, sandboxed code execution, and publishable result bundles. It also natively supports all three of our runtimes (vLLM, SGLang, llama-cpp-python) through simple environment variables.

Three custom evals

Pipeline latency (source). Tests whether multi-step LLM pipelines produce correct output when chained. A 2-step pipeline classifies the input then generates with a targeted prompt. A 4-step pipeline classifies, plans, executes, and verifies. If the blog's thesis is right (low latency enables chained calls), these pipelines should work at interactive speed on the fast models and degrade on the slow ones. Inspect tracks token usage and timing per step, so we can see where the pipeline spends its time.

Format compliance (source). Our quick test found that Bonsai outputs raw JSON while Gemma 4 wraps it in markdown fences. Two prompts is not enough to make that claim. This eval tests 10 prompts across JSON output, executable Python, yes/no answers, and numeric-only responses. The scorer checks whether the output is machine-parseable without post-processing.

Tool calling (source). Tests whether models can use tools (weather lookup, math calculation, database search) when given tool definitions. Simpler than BFCLv3 but directly relevant: if low-latency inference enables agentic workflows, the model needs to call tools correctly. Inspect's built-in tool support makes this a straightforward eval.

Setup

# Install
pip install inspect-ai

# Configure for your runtime
export LLAMA_CPP_PYTHON_BASE_URL=http://localhost:3001/v1    # Bonsai
export VLLM_BASE_URL=http://localhost:3000/v1                 # Gemma 4
export SGLANG_BASE_URL=http://localhost:3000/v1               # Qwen

# Run
inspect eval evals/pipeline_latency.py --model llama-cpp-python/bonsai
inspect eval evals/format_compliance.py --model vllm/google/gemma-4-E2B-it
inspect eval evals/tool_calling.py --model sglang/default

# View results (generates static HTML)
inspect view bundle --output results/

Results queued. These evals will run after the IFEval/GSM8K benchmarks complete. The eval files are in the repo and can be run independently against any OpenAI-compatible server.

What's next

Open questions

  • Gemma 4 MoE on H200. The 26B-A4B model with 3.8B active params per token is the most architecturally interesting for latency. It needs 48 GB VRAM at BF16. An H200 (141 GB HBM3e, 4.8 TB/s bandwidth) would give it room and speed. That is the next test I want to run.
  • Quantized Gemma 4. Running E4B at INT4 or FP8 would cut the model size and could significantly improve ms/tok. AWQ and GPTQ variants are starting to appear on HuggingFace but SGLang/vLLM support is still stabilizing.
  • Same-runtime comparison. Running Bonsai and Gemma 4 on the same engine would isolate the model-level latency difference from the runtime overhead. Neither engine supports both model formats today.
  • Longer outputs. All my tests used short outputs (10-48 tokens). Models behave differently on longer generations. Speculative decoding also becomes more relevant with longer outputs.
  • Bonsai vs Gemma 4 at same parameter count. Bonsai 4B vs Gemma 4 E4B (both ~4B effective) would be a fairer quality comparison.
