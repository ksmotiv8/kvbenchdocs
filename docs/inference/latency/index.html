Latency Changes Everything

Low inference latency is not just a speed improvement. It changes what systems you can build. I tested how low latency can go on a single GPU and found that the answer reshapes how I think about model selection, speculation, and the infrastructure underneath.

Thesis

Latency is a system design constraint, not a performance metric

At Momento, I have spent years obsessing over microsecond-level latency. Our caches and topics services operate at p99 latencies that most people would not believe are possible. That work gave me an intuition: latency is not just a number you optimize. It determines what you can build. Cross a threshold and entire system architectures that were previously impossible become viable.

I see the same dynamic playing out in LLM inference. At 2 seconds per response, you can build a chatbot. At 200ms, you can put an LLM in the request path of a web application. At 100ms, you can chain multiple inference calls inside a single user interaction without the user noticing. That is not a quantitative improvement. It is a qualitative one. New system designs become possible.

I wanted to find out where the boundary is today. I started on a single NVIDIA L4 because it is accessible and the constraints are real: 24 GB VRAM, ~300 GB/s memory bandwidth. If I have one GPU and one user, how fast can a response get? What are the real bottlenecks? Does speculative decoding help or hurt at this scale? Can aggressive quantization get latency low enough that inference disappears into the background of a larger system?

This is the starting point, not the destination. I want to see what happens on H200s with larger models, with 70B+ parameter counts and HBM3e bandwidth. But I wanted to understand the fundamentals first, on hardware where every millisecond of overhead is visible.

What I found surprised me. The fastest config was not the model I expected. The speculation strategy I thought would win did not. And a 1-bit model I tested as a curiosity ended up reframing how I think about model selection entirely.

Background

The tools

SGLang

SGLang is an open-source inference engine from UC Berkeley. What makes it relevant here is the control it gives you over latency-sensitive parameters. CUDA graph capture eliminates kernel launch overhead (microseconds that add up across hundreds of operations per token). RadixAttention reuses KV cache across requests with shared prefixes, which matters when your system prompt is the same every time. Continuous batching keeps the GPU busy without adding queuing delay to single requests.

The reason I use SGLang over vLLM or TGI for latency work is that it gives the lowest single-request latency I have measured across open-source engines. For throughput-optimized serving, the picture may be different. But for the single-user, single-request case I am interested in, SGLang consistently wins.

Speculative decoding

Speculative decoding tries to solve a fundamental problem: large models are slow because token generation is sequential, and each token requires reading the full weight matrix. The idea is to use a small, fast draft model to guess several tokens ahead, then verify all of them in a single forward pass of the main model. If the guesses are right, you get multiple tokens for the cost of one large-model pass.

The tradeoff is not obvious. The draft model adds its own latency. If it guesses wrong, you wasted that time. And the verification pass is slightly more expensive than a normal forward pass because it processes multiple candidate tokens. Whether speculation helps depends on the draft model's accuracy, its inference cost, and how many tokens you are generating. Two flavors tested here: EAGLE3 (tree-based attention, lightweight, explores multiple continuation paths simultaneously) and STANDALONE (a full smaller model as the draft, simpler but heavier).

1-bit Bonsai

1-bit Bonsai is a family of models by PrismML, released March 2026. The 8B variant is an end-to-end 1-bit quantized version of Qwen3-8B. Every weight in the model, including embeddings and the LM head, is stored as a single sign bit with one FP16 scale factor per group of 128 weights. That is 1.125 effective bits per weight, down from 16 in FP16. The result is a model that fits in 1.15 GB and runs on PrismML's custom llama.cpp fork. See the whitepaper for the full technical details.

One technical detail worth noting: single-request token generation is memory-bandwidth-bound. Each output token requires reading the full weight matrix from VRAM. This means that on the L4, smaller weights directly translate to faster tokens. That mechanism explains several of the results below, though it is not the whole story.

Why it matters

What becomes possible when latency drops

Before getting into the numbers, I want to explain why I care about latency this much. It is not about making chatbots feel snappy. It is about what you can build when inference is fast enough to use as a building block inside a larger system.

Prompt rewriting in the loop

One thing I built during this work was a prompt compiler (compile.sh) that classifies incoming prompts by complexity, rewrites them into more precise instructions, and then sends the rewritten prompt to the model for generation. The classification and rewriting are both LLM calls. If each call takes 2 seconds, the system is unusable. At 100-150ms per call, the user does not notice the extra step, and the output quality improves because the model receives a better prompt.

Classifier + router systems

With a fast small model, you can classify a request before deciding how to handle it. Simple questions get a cheap, fast answer. Complex questions get routed to a larger model or a multi-step pipeline. The classification itself is an inference call. If that call adds 2 seconds, you lose the latency savings on the simple path. If it adds 100ms, the routing overhead is negligible and you save compute on every simple request.

Multi-step inference without visible delay

Agentic systems that chain multiple LLM calls (plan, execute, verify, revise) are limited by the round-trip latency of each step. At 125ms per step, a 4-step pipeline completes in 500ms. That is interactive speed. At 1 second per step, the same pipeline takes 4 seconds and feels broken. The latency per step determines whether iterative refinement is a viable architecture or a theoretical one.

These are not hypothetical use cases. I used the first two in this project. The prompt compiler runs a 5-tier routing system: TRANSFORM (no LLM, deterministic edits), REWRITE (32 tokens, prompt cleanup), SINGLE_AGENT (48 tokens, technical rewrite), MULTI_AGENT (256 tokens, 4-agent plan), and REVIEW_LOOP (320 tokens, iterative design). The router decides which tier based on keyword matching and a complexity signal from ruflo. The simple tiers complete in under 150ms. That is fast enough to run in the critical path without the user feeling a delay.

The infrastructure has to keep up

There is a corollary to all of this that I think about constantly. If inference latency drops to 100ms, every other system in the pipeline has to match. A 100ms inference call followed by a 500ms vector search is still a 600ms round trip. Low latency inference will inevitably require ridiculously low latency RAG, vector search, and context retrieval. The model is no longer the bottleneck. The infrastructure around it is.

This is exactly the problem space I work in at Momento. Caches, topics, vector indexes, all operating at microsecond to low-millisecond latency. When inference gets fast enough to disappear into a pipeline, the question becomes whether your retrieval layer, your routing layer, and your context assembly can keep up. Most cannot. That gap is where the next wave of systems work will happen.

How to read this table

The natural instinct is to pick the row with the lowest total time. But the right choice depends on what you are building. There are three different strategies visible in the data:

Smallest model, no speculation (Qwen 1.5B). Lowest total latency at 125ms. Ideal for the routing and classification use cases above, where you need speed more than depth. The model is less capable, but if your prompt is well-shaped, a 1.5B model is often sufficient for short, constrained outputs. The tradeoff is correctness: on knowledge-heavy questions, it makes more mistakes.

Quantized large model, no speculation (Qwen 7B AWQ). 246ms total, better quality. If you can tolerate 250ms round trips, this gives you 7B-class answers without the complexity of a draft model. AWQ INT4 cuts the weight size enough that ms/tok is reasonable (18.9), though not as fast as the 1.5B.

Quantized large model + EAGLE3 (Llama AWQ). The best ms/tok (12.9) and lowest VRAM (6.2 GB), but 244ms total for short outputs. EAGLE3 shines when generating longer sequences because its per-token savings compound. For short outputs, the first-token latency dominates and speculation barely helps.

The STANDALONE draft with a 0.5B model added about 40ms of overhead. For 10-13 token outputs, that overhead was never recovered. The 1.5B draft was even worse: slower than no speculation at all. This is not a failure of the technique. It is a boundary condition. Speculation has a fixed cost per call, and when the generation is short, the fixed cost dominates.

Model selection depends on your latency budget. Under 150ms: use the smallest model that gives acceptable quality. Under 300ms: use a quantized larger model. Speculation helps only when output length justifies the draft model's fixed cost.

Quality: better than the 1.5B, but that is expected

I should be clear about what the quality comparison actually shows. Bonsai is an 8B model. Qwen is 1.5B. An 8B model should give better answers on knowledge tasks regardless of quantization. The meaningful comparison for quality would be Bonsai 1-bit 8B vs. Qwen 7B AWQ (4-bit) or Qwen 8B FP8 on the same prompts. I did not run that comparison. I plan to.

What I can say: the 1-bit 8B model did not show obvious quality degradation from quantization. When I asked about AWQ quantization, Qwen 1.5B said "Adaptive Weight Quantization" (wrong; the A stands for Activation-aware). Bonsai got it right. On CUDA graph capture, Bonsai described the DAG replay mechanism accurately while Qwen confused it with profiling. On code generation, they were roughly equal. Both produced the same 6k±1 prime checker algorithm.

PrismML's whitepaper reports Bonsai 8B scoring 70.5 average across six standard benchmarks (MMLU-Redux, MuSR, GSM8K, HumanEval+, IFEval, BFCLv3). That beats Llama 3.1 8B at FP16 (67.1) and LFM2 8B (69.6) at 1/14th the size. My side-by-side testing on the L4 was directionally consistent with those numbers, though I only tested 8 short-form prompts, not a full benchmark suite. I did not test reasoning, multi-step logic, or long-form generation, which are the places where aggressive quantization is most likely to show degradation.

Experiment 3

Gemma 4: Google's answer to low-latency inference

Gemma 4 launched on April 1st, 2026, and it is explicitly designed for the latency regime I care about. The family includes dense models (E2B at 2.3B effective params, E4B at 4.5B effective) and a Mixture-of-Experts model (26B total, only 3.8B active per token). Google is framing this as an edge and on-device architecture.

Two things caught my attention. First, the MoE variant: 128 experts with 8 active plus 1 shared per token. That means per-token compute is roughly 4B-class, but the model draws from 26B of stored knowledge. In theory, you get the latency of a small model with the capability of a large one. Second, the hybrid attention architecture: alternating local sliding window (512 tokens) and global attention layers, which reduces memory pressure on long contexts.

Compatibility note: Gemma 4 is days old. SGLang support is not merged yet (PR #21952 is open). I ran these tests on vLLM v0.19.0, which shipped Gemma 4 support on April 3rd. CUDA graphs are not yet working for Gemma 4 on either engine, so these numbers include extra kernel launch overhead that will improve.

Bonsai is 2.5x faster on both total latency and per-token speed. The gap is straightforward: Bonsai's 1-bit weights are 1.15 GB vs Gemma 4 E2B's 9.5 GB at BF16. On a bandwidth-limited GPU, that 8x weight size difference translates directly into generation speed.

But there is a caveat: Gemma 4 E2B uses Per-Layer Embeddings (PLE), which are large lookup tables that inflate total size (5.1B total vs 2.3B effective). These embeddings are not computed during generation, just looked up. A future FP8 or INT4 quantization of the E2B would shrink the non-PLE weights significantly and should close the ms/tok gap.

Quality comparison: different strengths

I ran quality prompts across factual knowledge, reasoning, code correctness, and instruction following. Gemma 4 models scored higher on knowledge. Bonsai scored higher on instruction following and format compliance.

Category Gemma 4 E2B Gemma 4 E4B Bonsai 1-bit 8B
Factual Knowledge 5/5 5/5 3/5
Reasoning 2/2 2/2 2/2
Code + Instruction 3/5 3/5 5/5
Total 10/12 10/12 10/12
ms/tok 20.8 40.5 8.3

All three models scored 10/12 overall, but with different profiles. Both Gemma 4 variants correctly identified AWQ as "Activation-aware Weight Quantization" while Bonsai said "Approximate" (wrong). At 2.3B and 4.5B effective parameters, that level of factual accuracy from the Gemma 4 family is notable.

Bonsai follows instructions more literally. When asked for raw JSON, Bonsai outputs {"name":"Qwen","language":"Chinese"}. Both Gemma 4 variants wrap it in a markdown code fence, which is helpful for humans but fails machine parsing. Same pattern with code: Gemma 4 wraps functions in triple-backtick blocks, Bonsai outputs executable code. For agentic and pipeline use cases where outputs feed into downstream systems, Bonsai's literal format compliance is more useful.

Interestingly, E4B did not score higher than E2B on quality despite having nearly double the effective parameters. The difference showed up in answer detail and verbosity, not in correctness. E4B's reasoning answers were longer and more thorough, but that cost significant latency (9.9s for a single calculation vs Bonsai's 2.1s).

Different models for different jobs. Gemma 4 is the better knowledge model per parameter. Bonsai is the better pipeline model: 4.7x faster than E4B, 2.5x faster than E2B, and its outputs are machine-parseable without post-processing. If your system chains LLM calls, format compliance matters as much as factual accuracy.

The MoE model I could not run

The most interesting Gemma 4 variant for latency is the 26B-A4B MoE: 128 experts, 8 active per token, only 3.8B active parameters. In theory, it should generate tokens at roughly 4B-class speed while accessing 26B of knowledge. But the full model is 48 GB at BF16 (all 128 experts must reside in VRAM even though only 8 fire per token). Even the smallest GGUF quantization (IQ2_M) is 9.3 GB. This model needs either a larger GPU (A100/H100/H200) or aggressive quantization in an engine that supports MoE quantization cleanly. Neither was available on the L4 during my testing window.

This is the test I most want to run next. If the MoE generates at ~4B speed with ~26B quality, it would be the strongest candidate for the "fast and smart" slot in a latency-optimized stack. The L4 is the wrong hardware for it. An H200 with 141 GB HBM3e would load the full BF16 model with room to spare.

Argument

Latency is a design constraint, not just a number

The point of this work is not "I made inference fast." It is that the latency numbers I measured change what systems are buildable. Let me connect the results back to the thesis.

At 125ms (Qwen 1.5B), an LLM call becomes a subroutine. You can put it in a request path, chain it with other calls, or use it as a classifier in a routing system without the user noticing. The prompt compiler I built uses this range for prompt rewriting and complexity classification. These are not the main inference call. They are infrastructure calls that happen before the main call, and they only work because they are fast enough to be invisible.

At 87-136ms (Bonsai 1-bit 8B), you get 8B quality at subroutine speed. That is the part I did not expect. The tradeoff I assumed was: fast but dumb (small model) or slow but smart (large model). Bonsai breaks that tradeoff by compressing the weights so aggressively that an 8B model moves through memory faster than a 1.5B model at conventional precision. The mechanism is memory bandwidth (1-bit weights are 14x less data to move per token), but the implication is architectural: you no longer have to choose between latency and capability for short-form tasks.

I should note a confound: Bonsai runs on llama.cpp while Qwen runs on SGLang. These runtimes have different overhead profiles (CUDA graphs, batching, warmup). Some of the speed difference could be runtime, not just model size. I did not isolate this. It would be worth running the same model on both runtimes to separate the effects.

Speculative decoding is a tool, not a default. It has a fixed cost per call (the draft model's forward pass) and a variable benefit (tokens saved by correct guesses). When the output is short, the fixed cost dominates and speculation hurts. When the output is long, the variable benefit compounds and speculation helps. The decision of whether to use it is a function of expected output length, not a universal configuration choice. I did not test the crossover point, but the data suggests it is somewhere above 20 tokens for STANDALONE and possibly lower for EAGLE3.

Gemma 4 shows what the next generation looks like. Even at E2B (2.3B effective parameters), Gemma 4 scored 5/5 on factual knowledge tasks where Bonsai (8.19B) scored 3/5. Google's Per-Layer Embeddings and hybrid attention architecture squeeze more capability per parameter than conventional dense models. The tradeoff today is VRAM: E2B uses 19.9 GB on vLLM for a 2.3B-effective model, making it impossible to run alongside other models on an L4. That will improve as quantized variants and runtime optimizations mature.

The Gemma 4 MoE variant (26B-A4B) is the one I most want to test. With 128 experts and only 3.8B active per token, it should generate at roughly 4B speed with 26B knowledge. On an H200 with 141 GB HBM3e and 4.8 TB/s bandwidth, this model could plausibly generate tokens in the sub-5ms/tok range. That would put 26B-class quality inside a 100ms round-trip budget with room to spare. I have not tested this yet.

Model selection is becoming a matrix, not a ranking. The right model depends on latency budget, quality requirements, output format needs, and VRAM constraints. Bonsai is fastest and most machine-parseable. Gemma 4 has better factual recall per parameter. SGLang Qwen is the most mature runtime integration. There is no single winner. The system designer has to match the model to the slot in the pipeline.

PrismML's whitepaper reports 44 tok/s for Bonsai 8B on an iPhone 17 Pro Max and 368 tok/s on a 4090 (I did not verify these). Google reports 7.6 tok/s for Gemma 4 E2B on a Raspberry Pi 5. Both demonstrate that useful inference on edge hardware is real, not theoretical. The gap between datacenter and edge is narrowing.

Caveats

What I did not test

I want to be honest about the limits of what I measured.

  • Batch throughput. All tests are single-request latency. Multi-user serving will have different characteristics. I did not test concurrent load.
  • Long outputs. SGLang benchmarks used short outputs (10-48 tokens). Speculative decoding may show better gains at longer output lengths. I plan to test this next.
  • Bonsai quality at scale. I tested 8 short-form prompts, not a full benchmark suite. No reasoning, multi-step, or long-form generation tests. PrismML's whitepaper evaluated across 10 benchmarks on H100 using vLLM and EvalScope. My L4 results are directionally consistent but not comprehensive.
  • Bonsai vs. same-size models. I compared Bonsai 8B against Qwen 1.5B, not against Qwen 7B AWQ or Qwen 8B FP8. The quality comparison is unfair to the 1.5B. A proper quality comparison at the same parameter count is the next thing I want to run.
  • Runtime confound. Bonsai runs on llama.cpp, Qwen runs on SGLang. These runtimes have different latency characteristics (CUDA graph support, batching, memory management). Some of the speed difference may be runtime, not model. I did not control for this.
  • Runtime compatibility. Bonsai requires PrismML's custom llama.cpp fork (PrismML-Eng/llama.cpp). No upstream llama.cpp, no SGLang, no vLLM support. If you need those runtimes, Bonsai is not an option today. Relying on a single-vendor fork in production carries its own risks.
  • Gemma 4 without CUDA graphs. Neither SGLang nor vLLM supports CUDA graph capture for Gemma 4 yet. My Gemma 4 latency numbers include extra kernel launch overhead. When CUDA graph support lands, expect 10-30% improvement on generation speed.
  • Gemma 4 E4B and MoE. The E4B (4.5B effective) crashed on FP8 load. The MoE (48 GB BF16) does not fit on the L4. I only tested the smallest variant (E2B). The larger variants are likely better on quality and the MoE is likely better on latency per quality-unit.
  • Other 1-bit approaches. I only tested PrismML's approach. BitNet and other 1-bit techniques exist. I did not compare them.
