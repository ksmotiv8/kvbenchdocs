Corpus Generator

Two-stage LLM pipeline for generating realistic clinical documents with exact token count control.

KV cache benchmarks need documents that behave like real workloads: varied vocabulary, mixed formatting, and the kind of token distributions that stress compression and cache access patterns in ways that synthetic hi hi hi text never will. We started with medical documents because clinical notes are a natural fit — they combine dense abbreviations, numeric lab values, structured sections, and narrative prose, all in a single document. That mix produces diverse token patterns and realistic tensor value distributions. We added legal documents as a second domain to validate that the generation pipeline generalizes: legal text has its own structure (clauses, cross-references, formal phrasing) and its own compression characteristics. Supporting multiple domains also lets us test whether benchmark results are sensitive to corpus choice, or whether the key finding — that realistic text behaves differently from synthetic text — holds across document types.

How It Works

Two-stage generation

The generator (corpus/build_corpus.py) connects to a running vLLM instance and works in two stages. Each domain (medical, legal) has its own prompt module with domain-specific Stage 1 and Stage 2 prompts. The pipeline itself is domain-agnostic: it loads the correct prompts based on --domain, generates scenarios, writes documents, and applies domain-specific noise.

1

Scenario Generation

Ask the LLM to define a realistic mix of document types for the chosen domain. Medical produces EHR document types; legal produces contracts, motions, briefs, and opinions.

→
2

Document Generation

For each scenario, prompt the LLM to write a complete document using domain-specific prompts, continuing and trimming until it hits the exact target token count.

Stage 1: Scenario generation

Each domain has its own Stage 1 prompt. The medical prompt asks for EHR document types; the legal prompt asks for litigation, transactional, regulatory, and advisory document types.

Medical Stage 1 Prompt 
You are a health informatics expert. Generate a JSON array of medical document
scenarios that represent the mix of clinical documents in a typical US hospital
EHR system. For each scenario provide:

- "scenario_id": short snake_case identifier
- "title": human-readable title
- "doc_type": the clinical document type (e.g. "ED Note", "Discharge Summary")
- "setting": clinical setting (e.g. "Emergency Department", "Primary Care")
- "weight": estimated proportion of all clinical documents (floats summing to 1.0)

Include at least 12 scenario types spanning inpatient, outpatient, emergency,
surgical, imaging, lab, specialty, and behavioral health settings.

Return ONLY the JSON array, no markdown fences, no explanation.
Legal Stage 1 Prompt 
You are a legal informatics expert. Generate a JSON array of legal document
scenarios that represent a diverse mix of legal documents produced across
different practice areas and jurisdictions. For each scenario provide:

- "scenario_id": short snake_case identifier
- "title": short descriptive title
- "doc_type": the legal document type (e.g. "Contract", "Motion", "Legal Brief")
- "setting": jurisdiction (e.g. "United States federal court")
- "context": 2-3 sentences describing the legal context
- "parties": the parties involved
- "weight": estimated proportion (floats summing to 1.0)

Include at least 10 scenario types spanning litigation, transactional,
regulatory, and advisory practice areas.

Return ONLY the JSON array, no markdown fences, no explanation.

The model returns a JSON array of scenarios with weights, normalized to sum to 1.0, then used to allocate documents proportionally. Legal scenarios include extra fields (context and parties) passed via **scenario unpacking, so each domain can define whatever scenario fields it needs.

Stage 2: Document generation

Each domain has its own Stage 2 template. The medical prompt asks for clinical document structure; the legal prompt asks for formal legal language, clauses, and internal references.

Medical Stage 2 Prompt 
Write a complete, realistic {doc_type} for a fictional patient.

Setting: {setting}
Scenario: {title}
Target length: approximately {token_target} tokens (write a thorough, detailed document).

Requirements:
- Invent realistic but fictional patient demographics, history, findings, and plans.
- Use proper medical terminology, abbreviations, and formatting conventions
  that would appear in an actual EHR system.
- Include all standard sections for this document type, but vary their ordering
  and depth naturally. Not every document follows the same rigid template.
- Include realistic vital signs, lab values with units and reference ranges,
  medication names with dosages, routes, and frequencies.
- Vary your writing style: some sections should be terse shorthand (e.g.
  "A&Ox3, NAD, PERRL"), others narrative paragraphs, others structured lists.
- Use a mix of abbreviations and spelled-out forms as a real clinician would.
- Do NOT include any meta-commentary. Output only the document itself.
- Write continuously until you have produced a detailed, complete document of
  approximately {token_target} tokens.
- This is document #{doc_number} for this scenario. Use a different patient,
  different chief complaint, different findings, and different clinical course
  than any other document in this batch.
Legal Stage 2 Prompt 
Write a complete, realistic legal document based on the scenario below.

Document type: {doc_type}
Title: {title}
Jurisdiction: {setting}
Context: {context}
Parties: {parties}
Target length: approximately {token_target} tokens.

Requirements:
- Write in formal legal language appropriate to the document type.
- Use dense legal phrasing typical of real legal writing.
- Use long paragraphs and complex sentence structures.
- Include internal references such as sections, clauses, definitions,
  or prior statements.

Structural variation requirements:
- Section headings should vary between documents.
- Section numbering style should vary (e.g. 1.1, Article I, Section A, (a)(i)).
- Avoid repeating identical templates or clause blocks.

Content realism:
- Use the fictional companies, agencies, or individuals from the scenario.
- Do not reference real cases, real statutes, or real people.
- Do not include meta-commentary. Output only the document itself.
- Write continuously until the document reaches approximately
  {token_target} tokens.
- This is document #{doc_number} for this scenario. Ensure it differs
  from other documents in structure, clause ordering, and specific terms.

Continuation loop

If the first response doesn't reach the target token count, the script sends the partial text back as assistant context along with a continuation prompt:

Prompt 
Continue writing the {doc_type} from exactly where you left off. Do NOT repeat
any content already written. Keep the same patient, same case, same formatting.
Write at least {remaining_tokens} more tokens of additional clinical detail —
expand on findings, add more lab results, imaging reads, nursing notes, consult
responses, or follow-up documentation. Output ONLY the continuation text, no
preamble.

This loop repeats until the accumulated text reaches the target (within 2% tolerance), at which point it's trimmed to the exact token count.

Reducing repetition and adding realism

LLM-generated documents tend to follow the same structural template when given the same prompt repeatedly. To reduce this effect, the generation prompts explicitly instruct the model to vary section ordering, mix terse shorthand with narrative paragraphs, and alternate between abbreviated and spelled-out terms.

After generation, an optional noise-injection step applies controlled text perturbations at a configurable rate (~12% of lines). The noise is domain-specific:

  • Medical noise: abbreviation variation (Blood Pressure / BP), case variation, punctuation variation (BP: 120/80 vs BP 120/80), section heading swaps (Chief Complaint / CC / CHIEF COMPLAINT), and whitespace variation.
  • Legal noise: section symbol insertion (Section 3.1 / §3.1), heading variation (Section / ARTICLE / Clause / Sec.), capitalized defined terms (agreement / Agreement), citation format variation (Section 5.2 / Sec. 5.2 / §5.2), and numbering style variation (1.1 / (a)).

The goal is to approximate the natural inconsistency found in real documents without changing the substantive content. Noise is applied after generation, then the document is re-trimmed to preserve the exact target token count.

The generation pipeline is domain-modular: each domain has its own prompt module under corpus/generators/ and its own noise function in corpus/noise.py. Adding a new domain means defining scenario prompts, document prompts, and noise rules. Documents are generated once, written to corpus/{domain}/ with a metadata.json manifest, and read from disk by the benchmark.

Token Count Accuracy

Solving the token count problem

The hi hi hi approach gets token counts right trivially: one word, one token, repeat N times. But when generating realistic clinical documents with an LLM, token count accuracy became a significant problem.

The initial approach: consistently oversized documents

Our first attempt asked the LLM to write documents of approximately 10,000 tokens, then relied on the API's completion_tokens field to track how many tokens had been generated. The results were consistently oversized:

Doc Type Actual Tokens Target Diff
0 ED Admission Note 14,333 10,000 +43.3%
1 ED Admission Note 12,598 10,000 +26.0%
2 ED Discharge Summary 14,180 10,000 +41.8%
5 Inpatient Order Set 19,574 10,000 +95.7%
6 Outpatient Visit 10,655 10,000 +6.6%
Mean (15 docs) 12,763 10,000 +27.6%

Root causes

1

Thinking tokens counted as content

Qwen3 models emit <think>...</think> reasoning blocks before generating text. The API's completion_tokens counts these thinking tokens, but the script strips them from the final document. A response reported as 8,000 tokens might only contribute 4,000 tokens of actual document text.

2

Character-ratio heuristics

The initial script tried to compensate by estimating "real" tokens using a characters-per-token ratio. This heuristic was unreliable because medical text has highly variable token density (abbreviations like "CBC" are 1 token, drug names like "acetaminophen" are 3–4 tokens).

3

No final verification

The script never tokenized the accumulated document text to verify the count. It just summed up estimated per-response counts and hoped for the best.

How we fixed it

The generator now loads the model's own HuggingFace tokenizer at startup and uses it for all counting:

Local tokenizer

transformers.AutoTokenizer gives exact token counts by tokenizing the actual document text, not by trusting API response metadata.

Thinking disabled at API level

Instead of stripping <think> blocks after the fact, the generator passes chat_template_kwargs: {"enable_thinking": False} via extra_body. This tells the model's chat template to skip reasoning entirely.

Full-text re-counting

After each continuation, the script tokenizes the entire accumulated document, not just the new chunk. This avoids drift from summing per-response estimates.

Encode-slice-decode trimming

Once the document reaches or exceeds the target, it's trimmed to exactly token_target tokens: encode to token IDs, slice to target length, decode back to text.

Token diversity ranges from 9.4% (medication reconciliation, heavy on repeated drug names and dosages) to 16.5% (lab report, many distinct test names and numeric values). By token diversity alone, all documents are roughly 640× more diverse than the hi hi hi baseline (0.02%). Token diversity does not directly translate to tensor-level diversity, but it does indicate that the model processes a wider range of lexical inputs than the synthetic benchmark.

Characters per token varies from 2.78 (medication reconciliation, with many short abbreviations and numbers) to 3.95 (discharge summary, with more natural prose). This variance is itself representative of real clinical text: different document types have different mixes of medical abbreviations, drug names, numeric lab values, and narrative text.

Usage

Get started

Generate, verify, and parse your own corpus with a few commands.

Prerequisites

Bash 
pip install openai transformers

A running vLLM instance (see Setting up the model).

Generate a corpus

Bash 
# 15 medical documents at exactly 10,000 tokens each (with noise)
python -m corpus.build_corpus \
    --domain medical \
    --documents 15 \
    --target-tokens 10000 \
    --noise \
    --api-base http://localhost:8000/v1

# 200 legal documents at 10,000 tokens each (with legal-specific noise)
python -m corpus.build_corpus \
    --domain legal \
    --documents 200 \
    --target-tokens 10000 \
    --noise \
    --api-base http://localhost:8000/v1

Output: corpus/{domain}/doc_0000.txt through doc_NNNN.txt, a benchmark-compatible JSONL file, and a metadata.json manifest. Generation takes ~7 minutes per document at 10K tokens on an L4 GPU.

The metadata manifest contains:

JSON 
{
  "domain": "medical",
  "documents": 15,
  "target_tokens": 10000,
  "generator_model": "Qwen/Qwen3-8B-FP8",
  "generation_date": "2025-03-15T...",
  "noise_enabled": true,
  "docs": [
    {"id": 0, "filename": "doc_0000.txt", "target_tokens": 10000,
     "actual_tokens": 10000, "tokens_before_noise": 10000,
     "scenario_id": "ed_initial_note", "doc_type": "ED Note"}
  ]
}

Verify token counts

Bash 
# Verify with the same tokenizer used for generation
python verify_tokens.py medical_docs_10000.jsonl

# Check how many tokens a different model's tokenizer sees
# (useful if you generated with Qwen but benchmark on a different model)
python verify_tokens.py medical_docs_10000.jsonl --model meta-llama/Llama-3-8B

# Include the benchmark prompt wrapper (as long_doc_qa.py sends it)
python verify_tokens.py medical_docs_10000.jsonl --include-prompt

# CSV output for spreadsheets
python verify_tokens.py medical_docs_10000.jsonl --csv > report.csv

Parse into individual documents

Bash 
# Default: writes docs/ and manifest.csv next to the JSONL file
python parse_corpus.py medical_docs_10000.jsonl

# Custom output directory
python parse_corpus.py medical_docs_10000.jsonl --output-dir ./parsed

# Skip the manifest CSV
python parse_corpus.py medical_docs_10000.jsonl --no-manifest

The individual .md files are formatted markdown that reads like authentic clinical notes, complete with patient demographics, vital signs, lab values, medication lists, imaging reports, and clinical assessments. (Note: parse_corpus.py works with the legacy JSONL format. The new corpus/build_corpus.py writes individual text files directly.)

Benchmark integration

The JSONL corpus is consumed directly by LMCache's long_doc_qa.py benchmark via the --corpus-file flag:

Bash 
python benchmarks/long_doc_qa/long_doc_qa.py \
    --model Qwen/Qwen3-8B-FP8 \
    --corpus-file /path/to/medical_docs_10000.jsonl \
    --num-documents 15 \
    --document-length 10000 \
    --output-len 100 \
    --repeat-count 1 \
    --repeat-mode tile \
    --max-inflight-requests 4

This replaces the default hi hi hi... generator with realistic medical documents, producing workloads that better represent the compression behavior and access patterns of production traffic.
