Background

KV caching and why it matters

Large language models process input text by computing key-value (KV) tensors at each attention layer. For a 10,000-token document on an 8B parameter model, the KV cache can consume hundreds of megabytes of GPU memory depending on layer count, hidden size, and KV precision. When multiple requests share the same document prefix, recomputing these tensors from scratch is pure waste.

LMCache solves this by offloading computed KV tensors to remote storage (Valkey, S3, or local disk) so subsequent requests can retrieve them instead of recomputing. This turns the expensive O(n²) prefill operation into a cache lookup, dramatically reducing time-to-first-token (TTFT) for repeated documents.

LMCache plugs directly into vLLM as a KV connector. When a request arrives, vLLM checks whether the KV cache for that prefix already exists in remote storage. If it does, the tensors are fetched and injected, skipping prefill entirely. If not, the tensors are computed normally and stored for future reuse.

Why we built this corpus

While optimizing KV cache offloading performance with Valkey as the remote backend, we ran into a benchmarking problem: the KV cache benchmarks we were using (long_doc_qa.py and multi_doc_qa.py) generate documents from synthetic hi hi hi sequences by default. LMCache does include a RAG benchmark that accepts real datasets and a multi-round QA benchmark that supports ShareGPT conversations, but the long-document benchmarks used for measuring offloading performance don't. The benchmark numbers looked great, but we couldn't trust them. We observed that compression ratios, tensor value distributions, and cache access patterns all changed materially when we moved from synthetic inputs to realistic text, and again when we moved from dummy weights to real trained weights. We needed a reproducible corpus of realistic documents to understand what performance actually looks like under representative conditions.

The Problem

The default benchmark document

LMCache ships with a long-document QA benchmark (long_doc_qa.py) for measuring KV cache offloading performance. When run without a corpus file, it generates documents like this:

Python 
warmup_prompts = [
    str(i) + " " + " ".join(["hi"] * args.document_length)
    for i in range(args.num_documents)
]

A "10,000-token document" is literally:

Output 
0 hi hi hi hi hi hi hi hi hi hi hi hi hi hi hi hi hi hi hi hi hi hi ...

The token count is accurate: the Qwen3-8B tokenizer encodes "hi" as a single token, so 10,000 repetitions produce 10,001 tokens. But the problem is what the tokens represent. The entire document contains only 2 unique tokens. This matters for KV cache benchmarks because:

Compression Behavior

KV cache connectors compress tensors (typically with Zstandard) before sending them to remote storage. In our experiments, repetitive inputs like hi hi hi yielded substantially higher compression ratios than realistic text. Benchmarks using only synthetic prompts can overestimate how much compression helps in a network-bound pipeline.

Activation Patterns

Transformers do not produce identical activations for repeated tokens: positional encoding, attention mixing, and residual connections mean each position's KV tensors are distinct. However, highly repetitive inputs often produce more structured activation patterns than diverse text, which can make the resulting tensors more compressible.

Cache Access Patterns

Real workloads involve documents with different prefixes, shared terminology, and overlapping but not identical content. The hi hi hi pattern creates a workload that shares nothing meaningful between documents except the same two tokens, which is unlikely to represent production traffic.

Chars/token 3.00 3.82

Benchmarking KV cache offloading with hi hi hi is like benchmarking a database with SELECT 1. The numbers come back fast, but they don't tell you how the system performs under realistic load.

Compression behavior with real model outputs

When we first started working with LMCache, we followed the tutorial setup and used --load-format dummy to skip real weight loading. With dummy weights, KV tensors compressed extremely well with Zstandard. The numbers were encouraging.

When we switched to Qwen3-8B-FP8 with real trained weights, in our measurements with Qwen3-8B-FP8 KV tensors, naive Zstandard compression provided only modest gains. Real model weights produce tensor values that behave more like high-entropy floating-point data. Where dummy weights might produce low-variance activations that compress easily, trained weights produce the kind of value distributions that general-purpose compressors struggle with.

Separately, we observed that the choice of input text also affected compression. Prompts made of repetitive text like hi hi hi compressed better than realistic documents. This is not because repeated tokens produce identical activations. Transformers still compute different values at each position due to positional encoding, attention context, and residual connections. But repetitive inputs often produce more structured activation patterns across positions, which can improve compressibility. Realistic documents with varied vocabulary, numeric values, abbreviations, and formatting tend to produce less structured activations and therefore weaker compression ratios.

The net effect: benchmarks using dummy weights or highly repetitive prompts can both overestimate how much compression reduces bytes on the wire. For a system where KV cache offloading is network-bound, that difference matters.

Making float tensors more compressible

Since naive Zstandard on raw FP16 tensor data wasn't giving us useful compression with real weights, we introduced bit shuffling as a pre-compression step.

Each FP16 value is 16 bits: 1 sign bit, 5 exponent bits, and 10 mantissa bits. When stored as raw bytes, the bit fields from many FP16 values are interleaved in memory, which often appears high-entropy to byte-oriented compressors. Bit shuffling rearranges the data so that corresponding bit positions from consecutive values are grouped together: all the sign bits in one block, all the exponent bits together, then the mantissa bits. Grouping similar bit positions across values exposes low-entropy regions that byte-oriented compressors like Zstandard can exploit effectively.

We used AVX2 intrinsics to keep the shuffle and unshuffle operations fast enough that CPU time did not become the new bottleneck. The goal was to reduce bytes on the wire without simply moving the constraint from network to CPU.

This is a practical tradeoff, not a silver bullet. Bitshuffle + Zstandard reduced transfer sizes enough to be useful on real model outputs, but the gains are more modest than what you'd see with dummy weights or synthetic inputs. That gap is exactly why we needed benchmarks that use realistic workloads.

Model Choice

Open weights for full reproducibility

We use Qwen/Qwen3-8B-FP8, an open-weight model, for both document generation and benchmarking. Using an open model means anyone can reproduce the corpus exactly (same tokenizer, same generation behavior, same KV cache tensor shapes) without depending on a proprietary API.

Qwen3-8B-FP8

FP8-quantized variant fits comfortably on a single NVIDIA L4 (24 GB) GPU. Weights download automatically from HuggingFace on first run.

Reproducible

Same tokenizer, same generation behavior, same KV cache tensor shapes. No proprietary API dependency, so anyone can reproduce the exact same corpus.

OpenAI-Compatible API

The vLLM server exposes an OpenAI-compatible API at localhost:8000/v1. The generation script connects to this endpoint directly.

Setting up the model

Bash 
# Install vLLM (0.15.0 was used for corpus generation)
pip install vllm

# Start the model server (single GPU, ~21 GB VRAM)
vllm serve Qwen/Qwen3-8B-FP8 \
    --tensor-parallel-size 1 \

curl http://localhost:8000/health

For the corpus included in this repo, we ran on an AWS g6.8xlarge instance (NVIDIA L4, 24 GB VRAM). Any GPU with 24+ GB VRAM should work. Generation takes ~7 minutes per 10K-token document.

How It Works

Two-stage generation

The generator (corpus/build_corpus.py) connects to a running vLLM instance and works in two stages. Each domain (medical, legal) has its own prompt module with domain-specific Stage 1 and Stage 2 prompts. The pipeline itself is domain-agnostic: it loads the correct prompts based on --domain, generates scenarios, writes documents, and applies domain-specific noise.

1

Scenario Generation

Ask the LLM to define a realistic mix of document types for the chosen domain. Medical produces EHR document types; legal produces contracts, motions, briefs, and opinions.

→
2

Document Generation

For each scenario, prompt the LLM to write a complete document using domain-specific prompts, continuing and trimming until it hits the exact target token count.

Stage 1: Scenario generation

Each domain has its own Stage 1 prompt. The medical prompt asks for EHR document types; the legal prompt asks for litigation, transactional, regulatory, and advisory document types. Here is the medical prompt:

Prompt 
You are a health informatics expert. Generate a JSON array of medical document
scenarios that represent the mix of clinical documents in a typical US hospital
EHR system. For each scenario provide:

- "scenario_id": short snake_case identifier
- "title": human-readable title
- "doc_type": the clinical document type (e.g. "ED Note", "Discharge Summary")
- "setting": clinical setting (e.g. "Emergency Department", "Primary Care")
- "weight": estimated proportion of all clinical documents (floats summing to 1.0)

Include at least 12 scenario types spanning inpatient, outpatient, emergency,
surgical, imaging, lab, specialty, and behavioral health settings.

Return ONLY the JSON array, no markdown fences, no explanation.

The model returns a JSON array of 12+ scenarios with weights. These weights are normalized to sum to 1.0, then used to allocate the requested number of documents across scenarios proportionally (e.g., if you ask for 15 docs and inpatient progress notes have weight 0.15, that scenario gets 2–3 documents).

The legal domain uses a different Stage 1 prompt that produces litigation, transactional, regulatory, and advisory scenarios:

Legal Stage 1 Prompt 
You are a legal informatics expert. Generate a JSON array of legal document
scenarios that represent a diverse mix of legal documents produced across
different practice areas and jurisdictions. For each scenario provide:

- "scenario_id": short snake_case identifier
- "title": short descriptive title
- "doc_type": the legal document type (e.g. "Contract", "Motion", "Legal Brief")
- "setting": jurisdiction (e.g. "United States federal court")
- "context": 2-3 sentences describing the legal context
- "parties": the parties involved
- "weight": estimated proportion (floats summing to 1.0)

Include at least 10 scenario types spanning litigation, transactional,
regulatory, and advisory practice areas.

Return ONLY the JSON array, no markdown fences, no explanation.

Legal scenarios include extra fields (context and parties) that are passed into the Stage 2 prompt via **scenario unpacking, so each domain can define whatever scenario fields it needs.

Stage 2: Document generation

Each domain has its own Stage 2 template. The medical prompt asks for clinical document structure; the legal prompt asks for formal legal language, clauses, and internal references. Here is the medical prompt:

Prompt 
Write a complete, realistic {doc_type} for a fictional patient.

Setting: {setting}
Scenario: {title}
Target length: approximately {token_target} tokens (write a thorough, detailed document).

Requirements:
- Invent realistic but fictional patient demographics, history, findings, and plans.
- Use proper medical terminology, abbreviations, and formatting conventions
  that would appear in an actual EHR system.
- Include all standard sections for this document type, but vary their ordering
  and depth naturally. Not every document follows the same rigid template.
- Include realistic vital signs, lab values with units and reference ranges,
  medication names with dosages, routes, and frequencies.
- Vary your writing style: some sections should be terse shorthand (e.g.
  "A&Ox3, NAD, PERRL"), others narrative paragraphs, others structured lists.
- Use a mix of abbreviations and spelled-out forms as a real clinician would.
- Do NOT include any meta-commentary. Output only the document itself.
- Write continuously until you have produced a detailed, complete document of
  approximately {token_target} tokens.
- This is document #{doc_number} for this scenario. Use a different patient,
  different chief complaint, different findings, and different clinical course
  than any other document in this batch.

The legal Stage 2 prompt uses the extra scenario fields ({context}, {parties}) to produce formal legal documents:

Legal Stage 2 Prompt 
Write a complete, realistic legal document based on the scenario below.

Document type: {doc_type}
Title: {title}
Jurisdiction: {setting}
Context: {context}
Parties: {parties}
Target length: approximately {token_target} tokens.

Requirements:
- Write in formal legal language appropriate to the document type.
- Use dense legal phrasing typical of real legal writing.
- Use long paragraphs and complex sentence structures.
- Include internal references such as sections, clauses, definitions,
  or prior statements.

Structural variation requirements:
- Section headings should vary between documents.
- Section numbering style should vary (e.g. 1.1, Article I, Section A, (a)(i)).
- Avoid repeating identical templates or clause blocks.

Content realism:
- Use the fictional companies, agencies, or individuals from the scenario.
- Do not reference real cases, real statutes, or real people.
- Do not include meta-commentary. Output only the document itself.
- Write continuously until the document reaches approximately
  {token_target} tokens.
- This is document #{doc_number} for this scenario. Ensure it differs
  from other documents in structure, clause ordering, and specific terms.

Continuation loop

If the first response doesn't reach the target token count, the script sends the partial text back as assistant context along with a continuation prompt:

Prompt 
Continue writing the {doc_type} from exactly where you left off. Do NOT repeat
any content already written. Keep the same patient, same case, same formatting.
Write at least {remaining_tokens} more tokens of additional clinical detail —
expand on findings, add more lab results, imaging reads, nursing notes, consult
responses, or follow-up documentation. Output ONLY the continuation text, no
preamble.

This loop repeats until the accumulated text reaches the target (within 2% tolerance), at which point it's trimmed to the exact token count.

Reducing repetition and adding realism

LLM-generated documents tend to follow the same structural template when given the same prompt repeatedly. To reduce this effect, the generation prompts explicitly instruct the model to vary section ordering, mix terse shorthand with narrative paragraphs, and alternate between abbreviated and spelled-out terms.

After generation, an optional noise-injection step applies controlled text perturbations at a configurable rate (~12% of lines). The noise is domain-specific:

  • Medical noise: abbreviation variation (Blood Pressure / BP), case variation, punctuation variation (BP: 120/80 vs BP 120/80), section heading swaps (Chief Complaint / CC / CHIEF COMPLAINT), and whitespace variation.
  • Legal noise: section symbol insertion (Section 3.1 / §3.1), heading variation (Section / ARTICLE / Clause / Sec.), capitalized defined terms (agreement / Agreement), citation format variation (Section 5.2 / Sec. 5.2 / §5.2), and numbering style variation (1.1 / (a)).

The goal is to approximate the natural inconsistency found in real documents without changing the substantive content. Noise is applied after generation, then the document is re-trimmed to preserve the exact target token count.

The generation pipeline is domain-modular: each domain has its own prompt module under corpus/generators/ and its own noise function in corpus/noise.py. Adding a new domain means defining scenario prompts, document prompts, and noise rules. Documents are generated once, written to corpus/{domain}/ with a metadata.json manifest, and read from disk by the benchmark.

Token Count Accuracy

Solving the token count problem

The hi hi hi approach gets token counts right trivially: one word, one token, repeat N times. But when generating realistic clinical documents with an LLM, token count accuracy became a significant problem.

The initial approach: consistently oversized documents

Our first attempt asked the LLM to write documents of approximately 10,000 tokens, then relied on the API's completion_tokens field to track how many tokens had been generated. The results were consistently oversized:

Doc Type Actual Tokens Target Diff
0 ED Admission Note 14,333 10,000 +43.3%
1 ED Admission Note 12,598 10,000 +26.0%
2 ED Discharge Summary 14,180 10,000 +41.8%
5 Inpatient Order Set 19,574 10,000 +95.7%
6 Outpatient Visit 10,655 10,000 +6.6%
Mean (15 docs) 12,763 10,000 +27.6%

Root causes

1

Thinking tokens counted as content

Qwen3 models emit <think>...</think> reasoning blocks before generating text. The API's completion_tokens counts these thinking tokens, but the script strips them from the final document. A response reported as 8,000 tokens might only contribute 4,000 tokens of actual document text.

2

Character-ratio heuristics

The initial script tried to compensate by estimating "real" tokens using a characters-per-token ratio. This heuristic was unreliable because medical text has highly variable token density (abbreviations like "CBC" are 1 token, drug names like "acetaminophen" are 3–4 tokens).

3

No final verification

The script never tokenized the accumulated document text to verify the count. It just summed up estimated per-response counts and hoped for the best.

How we fixed it

The generator now loads the model's own HuggingFace tokenizer at startup and uses it for all counting:

Local tokenizer

transformers.AutoTokenizer gives exact token counts by tokenizing the actual document text, not by trusting API response metadata.

Thinking disabled at API level

Instead of stripping <think> blocks after the fact, the generator passes chat_template_kwargs: {"enable_thinking": False} via extra_body. This tells the model's chat template to skip reasoning entirely.

Full-text re-counting

After each continuation, the script tokenizes the entire accumulated document, not just the new chunk. This avoids drift from summing per-response estimates.

Encode-slice-decode trimming

Once the document reaches or exceeds the target, it's trimmed to exactly token_target tokens: encode to token IDs, slice to target length, decode back to text.

Usage

Get started

Generate, verify, and parse your own corpus with a few commands.

Prerequisites

Bash 
pip install openai transformers

A running vLLM instance (see Setting up the model above).

Generate a corpus

Bash 
# 15 medical documents at exactly 10,000 tokens each (with noise)
python -m corpus.build_corpus \
    --domain medical \
    --documents 15 \
    --target-tokens 10000 \
    --noise \
    --api-base http://localhost:8000/v1

# 200 legal documents at 10,000 tokens each (with legal-specific noise)
python -m corpus.build_corpus \
    --domain legal \
    --documents 200 \
    --target-tokens 10000 \
    --noise \
    --api-base http://localhost:8000/v1

Output: corpus/{domain}/doc_0000.txt through doc_NNNN.txt, a benchmark-compatible JSONL file, and a metadata.json manifest. Generation takes ~7 minutes per document at 10K tokens on an L4 GPU.

The metadata manifest contains:

JSON 
{
  "domain": "medical",
  "documents": 15,
  "target_tokens": 10000,
  "generator_model": "Qwen/Qwen3-8B-FP8",
  "generation_date": "2025-03-15T...",
  "noise_enabled": true,
  "docs": [
    {"id": 0, "filename": "doc_0000.txt", "target_tokens": 10000,
     "actual_tokens": 10000, "tokens_before_noise": 10000,
     "scenario_id": "ed_initial_note", "doc_type": "ED Note"}
  ]
}

Verify token counts

Bash 
# Verify with the same tokenizer used for generation
python verify_tokens.py medical_docs_10000.jsonl

# Check how many tokens a different model's tokenizer sees
# (useful if you generated with Qwen but benchmark on a different model)
python verify_tokens.py medical_docs_10000.jsonl --model meta-llama/Llama-3-8B

# Include the benchmark prompt wrapper (as long_doc_qa.py sends it)
python verify_tokens.py medical_docs_10000.jsonl --include-prompt

# CSV output for spreadsheets
python verify_tokens.py medical_docs_10000.jsonl --csv > report.csv

Parse into individual documents

Bash 
# Default: writes docs/ and manifest.csv next to the JSONL file
python parse_corpus.py medical_docs_10000.jsonl

# Custom output directory
python parse_corpus.py medical_docs_10000.jsonl --output-dir ./parsed

# Skip the manifest CSV
python parse_corpus.py medical_docs_10000.jsonl --no-manifest

The individual .md files are formatted markdown that reads like authentic clinical notes, complete with patient demographics, vital signs, lab values, medication lists, imaging reports, and clinical assessments. (Note: parse_corpus.py works with the legacy JSONL format. The new corpus/build_corpus.py writes individual text files directly.)

Benchmark integration

The JSONL corpus is consumed directly by LMCache's long_doc_qa.py benchmark via the --corpus-file flag:

Bash 
python benchmarks/long_doc_qa/long_doc_qa.py \
    --model Qwen/Qwen3-8B-FP8 \
    --corpus-file /path/to/medical_docs_10000.jsonl \
    --num-documents 15 \
    --document-length 10000 \
    --output-len 100 \
    --repeat-count 1 \
    --repeat-mode tile \
    --max-inflight-requests 4

This replaces the default hi hi hi... generator with realistic medical documents, producing workloads that better represent the compression behavior and access patterns of production traffic.

Benchmark Tool

bench_kvcache.py

An improved KV cache benchmark that addresses limitations in LMCache's built-in long_doc_qa.py. Supports loading the medical corpus directly, skipping warmup phases, and reporting detailed TTFT percentiles.

How it differs from LMCache benchmarks

LMCache ships 7 benchmark tools. The long-document and multi-document benchmarks generate documents using repeated "hi" tokens by default (2 unique tokens, 0.02% diversity). bench_kvcache.py replaces this with realistic content:

Corpus file support

--corpus-file loads documents directly from the JSONL medical corpus instead of generating synthetic hi hi hi padding. Also supports 5 built-in text styles (mixed, technical, legal, conversation, narrative) when no corpus is provided.

Independent concurrency

Separate --warmup-concurrency and --query-concurrency controls. LMCache uses a single setting for both phases, making it impossible to warm the cache serially while querying at high concurrency.

Skip warmup

--skip-warmup and --skip-pre-warmup flags let you benchmark against an already-warm cache. LMCache always runs warmup with no way to skip it.

Detailed metrics

Where LMCache reports only a trimmed mean TTFT, bench_kvcache.py reports a full statistical breakdown per phase:

Metric bench_kvcache.py LMCache long_doc_qa.py
TTFT min / max Yes No
TTFT p50 / p95 / p99 Yes No
Trimmed mean Yes Yes
Throughput (prompts/s) Yes No
CSV with config metadata Append-mode Separate files
Visualization Histogram + scatter + CDF Gantt chart
Timer precision perf_counter time.time

Realistic cache miss simulation

The --hit-ratio flag controls what fraction of query-round requests are cache hits. Misses use entirely new documents with realistic vocabulary, not the same prompt with a random number prepended (which is what LMCache does). This produces more realistic mixed-workload benchmarks.

Running the benchmark

Bash 
# Run with the medical corpus
python bench_kvcache.py \
    --model Qwen/Qwen3-8B-FP8 \
    --corpus-file medical_docs_10000.jsonl \
    --query-concurrency 4 \
    --max-tokens 100 \
    --csv results.csv

# Skip warmup (cache already populated from a previous run)
python bench_kvcache.py \
    --model Qwen/Qwen3-8B-FP8 \
    --corpus-file medical_docs_10000.jsonl \
    --skip-warmup \
    --query-concurrency 8

# Simulate 75% cache hit ratio
python bench_kvcache.py \
    --model Qwen/Qwen3-8B-FP8 \
    --corpus-file medical_docs_10000.jsonl \
    --hit-ratio 0.75 \
    --visualize results.png
