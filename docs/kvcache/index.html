KV Cache for LLM Inference

Prefill is turning inference into a distributed systems problem. KV cache is the object at the center of it.

I have been fascinated by prefill. Not the part everyone talks about (making generation faster), but the part that is underappreciated: prefill is the work that turns inference into a distributed systems problem.

Most conversations about LLM serving treat the model as the interesting part. The model is not the interesting part. The interesting part is that prefill and decode have fundamentally different compute profiles, and the moment you separate them, you need something to carry state across the boundary. That something is KV cache. It stops being an implementation detail and starts being a systems primitive.

This is the same structural shift that happened in data infrastructure when Snowflake separated storage and compute. Before the separation, compute and storage were colocated and scaled together. After the separation, they became independently addressable resources connected by a transfer layer. Disaggregated inference is following the same arc. Prefill nodes produce KV cache. Decode nodes consume it. The transfer layer between them is where the systems work happens.

That is what excites me about the inference problem right now. It is not a model problem. It is a distributed systems problem with real tradeoffs in placement, caching, transfer, eviction, and capacity planning. The kind of problem where getting the systems layer right compounds across every request.

This series explores that problem from multiple angles.
